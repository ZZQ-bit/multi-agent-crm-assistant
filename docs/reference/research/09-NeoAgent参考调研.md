# NeoAgent 参考调研

> 文档状态：外部产品调研资料。
>
> 本文用于保留竞品信息和设计来源，不定义当前产品范围、页面结构、Chatflow 或验收标准。

## 1. 调研背景

本文基于 2026-06-18 可公开检索到的销售易官方资料，整理 NeoAgent / NeoAgent 2.0 的产品定位、关键能力、与当前 `多Agent智能助手` 方案的参考关系。

本调研重点不是复述宣传口径，而是回答三个问题：

1. `NeoAgent` 到底做成了什么形态。
2. 它哪些设计对我们当前 `docs/06` 与 `docs/08` 有直接参考价值。
3. 哪些能力适合借鉴，哪些不应直接照搬。

---

## 2. 快速结论

### 2.1 一句话判断

`NeoAgent` 最值得参考的不是“六大 Agent”的命名本身，而是它把 CRM、企业知识、非结构化互动数据、业务动作、人工确认、可观测运营这些能力，收敛成了一个“可执行的业务工作台”。

### 2.2 与我们项目最相关的 5 个参考点

1. 不是聊天机器人，而是业务工作台。
2. 强依赖 CRM 上下文与多源数据，不做脱离业务对象的空对话。
3. 结果不是只给答案，而是给“下一步动作 + 任务/写回/执行闭环”。
4. 支持多 Agent 分工，但强调过程可解释、可观测、可调试。
5. 强调“关键动作需人工确认”，避免 AI 越权自动执行。

## 3. NeoAgent 是什么

### 3.1 2025 年版本：AI CRM + 6 类业务 Agent

销售易在 2025-03-24 发布的官方文章中，把 NeoAgent 定义为“中国首款 AI CRM”，并给出 6 类垂直业务智能体：

- 销售助理 Agent
- 销售经理 Agent
- 销售教练 Agent
- 营销 Agent
- 客服 Agent
- 分析师 Agent

官方同时强调，NeoAgent 背后不是单一对话能力，而是：

- `NeoAgent 平台`：用于定义 Prompt 模板、Agent 主题、Action
- `Customer Data Cloud`：作为统一客户数据平台和 AI 数据底座

## 4. 官方资料里最值得我们关注的能力

### 4.1 CRM 上下文不是可选项，而是核心输入

销售易官方反复强调，NeoAgent 不只是读结构化 CRM 字段，还要整合：

- 客户与商机数据
- 拜访录音
- 邮件
- 企微聊天
- 活动记录
- 工单
- 企业知识库
- 外部网络情报

这和我们 `docs/06`、`docs/08` 中“中间回合流承载业务判断，必要时轻量展示客户/商机对象”的方向一致。右侧 CRM 上下文区曾作为参考方案，但当前 P0 已收敛为智能助手结构，不再做固定第三栏。

对我们的启发：

- 多Agent智能助手 不能只围绕“提问-回答”设计。
- 要把“当前客户/商机/最近跟进/跟进计划/附件材料”视为一等输入，但不必把它们常驻展示成第三栏。
- 后续接真实接口时，要优先接入可沉淀业务语义的对象，而不是只接聊天记录。

### 4.2 从结论型 AI，走向动作型 AI

销售易在 `NeoAgent 2.0` 和 `智慧销售工作台` 相关资料里，反复强调 AI 不只是给建议，还会：

- 自动生成客户洞察报告
- 推荐下一步行动
- 自动创建任务
- 自动整理活动记录
- 推动商机进入下一阶段
- 自动更新预测

对我们的启发：

- `多Agent智能助手` 的价值不能停留在“生成评审报告”。
- 应优先让输出落到“可执行动作”和“可确认草稿”上。
- 这正好对应 `docs/06` 中已经定义的写回草稿、人工确认、回合闭环。

### 4.3 多 Agent 不是目的，业务分工才是目的

销售易 2025 年版本重点在六类 Agent，2026 年版本则更强调：

- 销售 Agent
- 服务 Agent
- 业务专项子 Agent
- Agent 开发平台

说明它的重心从“列出很多 Agent”逐步转向“围绕业务目标组合智能执行能力”。

对我们的启发：

- 我们不需要执着复制“六大 Agent”。
- 更适合沿用自己文档中的分工：`sales / finance / delivery / legal / coordinator`。
- 页面上要展示的是“本轮用了哪些业务角色，各自输出了什么”，而不是把 Agent 本身产品化成资产管理页。

### 4.4 过程可解释、可调试、可观测

销售易官方明确提到：

- Prompt Builder
- Action 配置
- 可视化调试
- Agent 思考链路与工具调用展示
- 智能体运营看板
- 实时 Token 流量监控
- 完整运营日志

对我们的启发：

- 我们前台页面不应该暴露真实思维链，但后台一定要有过程日志和可调试能力。
- `docs/06` 里“业务判断过程可见，但不展示模型原始思维链”的边界是对的。
- 后续如果接 Dify / 自研 orchestrator，最好同时设计：
  - 用户可见的业务过程摘要
  - 管理员可见的运行日志
  - Prompt / 工具 / 数据源的调试视图

### 4.5 人工确认是关键控制点

销售易在 `智慧销售工作台` 文章里明确提到：关键任务需要销售审批，AI 只自动执行已授权动作。

这与我们 `docs/06` 和 `docs/08` 中的设计完全同向：

- 写回草稿先展示
- 用户可编辑
- 用户确认后再写入
- 高风险动作不自动落库

这是一个非常重要的产品判断：  
真正可落地的 B 端 Agent，不是“全自动”，而是“业务闭环里的半自动 + 可追责 + 可确认”。

## 5. 和我们 `docs/06`、`docs/08` 的对照

### 5.1 高度一致的部分

#### A. 工作台而非聊天页

我们文档：

- 当前 P0 收敛为“左历史 + 中间对话”的智能助手结构
- 历史会话常驻
- 中间按回合组织
- 客户/商机只在本轮回答需要时轻量提示

NeoAgent 参考：

- 官方已从“AI 助手”演进到“智慧销售工作台”
- AI 深度融入销售日常动作，而不是单独悬浮在业务外

#### B. 结果之后必须接动作

我们文档：

- 写回草稿
- 跟进记录 / 跟进计划
- 人工确认

NeoAgent 参考：

- 行动建议
- 一键转任务
- 自动沉淀记录
- 关键动作需审批

#### C. 上下文驱动的业务分析

我们文档：

- 缺上下文时进入待确认态
- 商机评审依赖客户/商机对象

NeoAgent 参考：

- 基于 CRM、知识库、互动数据、外部情报形成洞察
- 企微会话中也持续绑定客户语境和业务对象

### 5.2 我们可以进一步补强的部分

#### A. 主动提醒与定时刷新

销售经理 Agent 官方资料提到：

- 阶段变化时自动刷新洞察
- 30/60/90 天自动刷新客户健康度
- 主动提醒客户变化

对我们可补强：

- 当前文档更偏“用户发起一次评审”
- 后续可以扩展为“系统主动提示风险变化 / 推荐重新评审”

#### B. 多入口协同

销售易 2026 年强调：

- CRM 内工作台
- 企业微信 / WorkBuddy 等轻入口

对我们可补强：

- 当前先做一级菜单入口没问题
- 但结构上要为“从客户页、商机页、IM/企微消息、提醒卡片跳入”预留 query 和上下文注入机制

#### C. 可观测运营

当前我们的文档更关注前台页面与 mock 流程，还没有系统化定义：

- 智能体调用统计
- Prompt 调整效果
- 失败率 / 幻觉率观测
- 关键写回动作审计

这些正是 NeoAgent 2.0 很值得参考的“平台层能力”。

## 6. 对当前 多Agent智能助手 最有价值的借鉴建议

### 建议 1：保持“三栏工作台 + 回合流”路线，不要退回纯聊天页

NeoAgent 的产品演进说明，真正能承载业务价值的 Agent 产品，最终都会回到：

- 左边：会话 / 任务索引
- 中间：业务回合与结果
- 右边：对象上下文与资料

这正好验证了 `docs/08` 的方向。

### 建议 2：把“下一步动作”做成一等输出，不只是附属建议

除了总体结论与风险项，建议在结果区固定保留：

- 下一步动作
- 推荐负责人
- 推荐时间
- 是否生成跟进记录 / 跟进计划草稿

这会让页面从“会分析”变成“能推进业务”。

### 建议 3：把“过程可见”严格限定在业务过程，不暴露模型思维链

NeoAgent 的官方材料提到可视化调试，但那更像平台/管理员能力。  
对用户端页面，仍应坚持我们现在的原则：

- 展示任务识别、上下文读取、专项检查、风险发现
- 不展示 Prompt、原始工具请求、内部推理草稿

### 建议 4：尽早预留“主动触发”的能力模型

即使 P0 仍是用户手动提问，数据结构上也建议预留：

- 风险变化提醒
- 阶段变化重评审
- 周报式自动复盘
- 会话总结自动生成草稿

这会让后续演进更像 NeoAgent 2.0，而不是停留在“问一次答一次”。

### 建议 5：把“写回审计”视为平台能力，不只是前端按钮

后续真实接入时，建议补一层：

- 谁触发了草稿
- 草稿基于哪些上下文生成
- 谁编辑过
- 谁确认写入
- 写入了哪些字段

这类审计能力对企业落地很关键，也是我们与“AI 演示页”拉开差距的地方。

## 7. 不建议直接照搬的地方

### 7.1 不要照搬销售易的 Agent 命名与矩阵

原因：

- 销售易服务的是通用 CRM 市场
- 我们当前场景更聚焦 `多Agent智能助手 / 商机评审 / 风险判断 / 写回确认`

所以更适合保留我们自己的业务分工模型。

### 7.2 不要一开始就追求“自动执行一切”

NeoAgent 官方虽然强调主动执行，但也反复强调：

- 授权边界
- 审批确认
- 不越权

对我们来说，先把“建议 + 草稿 + 确认 + mock 回写”跑顺，比追求全自动更重要。

### 7.3 不要把平台能力和前台页面能力混在一起

NeoAgent 有两层：

- 用户看的业务工作台
- 管理员/实施顾问用的 Agent 平台能力

我们当前 P0 更适合先完成前者，再逐步补后者。

## 9. 最后判断

如果把 NeoAgent 的参考价值压缩成一句话：

> 它证明了“企业级多 Agent 产品”的正确打开方式，不是做一个更会聊天的机器人，而是做一个吃透 CRM 上下文、能生成动作闭环、且受人工确认约束的业务工作台。

这与我们当前 `多Agent智能助手` 的产品方向是同频的。  
后续最值得吸收的不是“长得像销售易”，而是把下面四件事做扎实：

- 上下文绑定
- 回合化分析
- 动作草稿与确认
- 平台级可观测与审计
