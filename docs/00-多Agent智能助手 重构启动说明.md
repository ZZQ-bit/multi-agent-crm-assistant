# 多Agent智能助手 重构启动说明

## 1. 文档目的

本文档合并了当前阶段的“问题盘点”和“整体调整路线”，作为正式文档重构完成前的临时基线。

在 `00-文档索引.md` 和 `01-多Agent智能助手 PRD.md` 建立前，后续协作应优先参考本文档，避免继续从旧文档中各取一段，造成规划分叉和上下文浪费。

## 2. 当前结论

- 项目已经具备可继续打磨的基础：CordysCRM 后端、前端 多Agent智能助手 页面、Dify Chatflow YAML、知识库规则和部分协议验证脚本。
- 当前最大问题不是“有没有材料”，而是“当前唯一基线不清楚”：文档多、版本混杂、PRD 缺失，容易让后续 Agent 误读项目重点。
- 多Agent智能助手 的核心价值还没有完全跑通：Chatflow 已接入对话接口，但还没有充分使用 CRM 工具能力完成“查询业务数据 -> 多 Agent 分析 -> 生成建议 -> 人工确认 -> 写回 CRM”的闭环。
- 下一步应先做文档重构和 P0 业务闭环确认，再进入大规模功能实现。

## 3. 问题盘点与优先级

### P0 问题

#### P0-01 文档体系缺少当前唯一基线

- `docs/` 下文档数量较多，包含方案说明、原型说明、实现记录、验证报告、历史 PRD、参考资料等。
- 旧文档与新规划并存，后续 Agent 容易把历史材料误判为当前方案。
- 需要先建立“当前文档索引”和“归档规则”，明确哪些文档是当前依据，哪些仅作为参考。

#### P0-02 多Agent智能助手 PRD 主文档缺失

- 当前 `01-项目方案说明.md` 更像项目总览，不足以承担产品主干。
- `cordyscrm-v1.7.1-original-prd.md` 是 CordysCRM 底座 PRD，不是 多Agent智能助手 的产品 PRD。
- 多Agent智能助手 需要独立 PRD，明确目标用户、业务场景、核心流程、功能边界、验收标准和演示范围。

#### P0-03 Chatflow 工具能力已接入，当前重点转为只读演示稳定性

- 当前以 `chatflows/ai-deal-desk-v3.example.yml` 为准；该 Chatflow 已改为 Planner、证据台账、按需多 Agent 和统一协议适配的只读链路。
- CRM 查询、知识库、外部情报和附件证据都作为证据来源进入统一证据台账。
- 当前阶段不再追求自动写回 CRM；用户要求保存或写入时，只生成可复制草稿，真实写回作为后续独立阶段处理。

#### P0-04 CRM 能力存在，但未封装成 多Agent智能助手 可用工具链

- 后端已有客户、商机、跟进、搜索等业务控制器。
- `runtime/cordys/cordys-crm.properties` 中存在 `mcp.embedded.enabled=true`，说明项目已有 MCP 方向的基础。
- 当前 MCP 暴露能力偏配置查询，尚未形成 多Agent智能助手 所需的客户查询、商机查询、跟进记录查询、跟进计划写入等业务工具集。

#### P0-05 P0 业务闭环尚未固定为验证主线

- 项目同时包含快速问答、复杂评审、沟通生成、跟进计划等多条能力线。
- 这些能力都重要，但当前阶段需要先选定一条最能体现作品集价值的 P0 主线。
- 建议 P0 主线固定为“复杂商机评审”：销售选择商机后，AI 查询 CRM 数据，多 Agent 分析，并生成可确认写回的建议或跟进计划。

#### P0-06 验证体系缺失

- 当前已有 smoke test 和部分协议验证，但主要验证本地流程、Markdown 输出或协议结构。
- 缺少围绕 P0 主线的端到端验证：真实或准真实 CRM 数据输入、Chatflow 工具调用、多 Agent 输出、前端展示、人工确认、CRM 写回。

### P1 问题

#### P1-01 页面信息架构口径不完全一致

- 该问题已在 `docs/01-多Agent智能助手 PRD.md` 中收敛。
- P0 页面结构以当前双栏工作台为准：左侧历史会话 + 中间对话主工作区。
- P0 不做固定右侧业务栏；业务对象通过输入框绑定、路由对象、会话记忆和 CRM 详情页承载；多结果通过 Markdown 文本澄清。

#### P1-02 演示数据和页面文案需要统一清理

- 代码、文档、测试中存在 `mock`、内部状态名、技术标签、过期示例客户等内容。
- 原型页面必须只展示真实用户会看到的业务内容，不应出现内部调试词、技术变量名或设计说明。

#### P1-03 前端仍存在真实接口与 mock fallback 双轨

- Dify Provider 和降级逻辑可以帮助演示不断线，但也可能掩盖真实联调失败。
- 后续需要明确演示模式、开发模式和验收模式的差异。

#### P1-04 写回 CRM 当前暂停，协议保留为后续能力

- 项目已有“人工确认后写回”的产品方向，但当前 V3 只读版不调用写回工具。
- `writeback` 相关协议保留为兼容字段，后续如恢复真实写回，需要单独确认真实写入动作、确认 UI、失败处理、撤销或审计口径。

#### P1-05 知识库规则未与新 PRD 和工具化 Chatflow 对齐

- 知识库已有业务规则基础，但应重新对齐 P0 主线和多 Agent 分工。
- 后续需要区分：哪些规则进入 Dify 知识库，哪些规则进入 Agent Prompt，哪些规则进入 PRD 或验收清单。

#### P1-06 后端缺少 多Agent智能助手 专用编排层

- 后端目前更像 CordysCRM 业务 API 和协议适配。
- 若要支撑 多Agent智能助手，应补充面向 AI 工具调用的薄封装层，避免 Chatflow 直接理解过多 CRM 底层接口细节。

### P2 问题

#### P2-01 作品集入口和 README 表达不足

- 项目需要一个面向评审者的入口，解释它是什么、解决什么问题、如何演示、亮点在哪里。

#### P2-02 历史材料占用上下文

- 参考模板、历史 PRD、旧路线和实现记录仍有价值，但不应与当前文档混放。
- 建议统一放入 `docs/archive/` 或 `docs/reference/`，并在索引中说明其用途。

#### P2-03 高保真原型与实际前端可能继续分叉

- 原型可以先行，但最终应以 PRD 和验收脚本约束前端实现，避免出现“图很好看，但演示闭环跑不通”的情况。

## 4. 调整路线

### 阶段 0：问题盘点与现状审计

目标：确认当前项目的真实问题清单，避免只解决已经想到的几个点。

产出：

- 问题盘点与优先级。
- 当前文档、Chatflow、CRM 接口、前端页面、知识库、测试脚本的审计结论。

当前状态：已完成初版盘点，本文档作为合并后的启动基线。

### 阶段 1：文档重构与当前基线收敛

目标：减少文档数量，建立清晰的当前文档体系。

建议收敛为：

- `docs/00-文档索引.md`
- `docs/01-多Agent智能助手 PRD.md`
- `docs/02-知识库与业务规则设计.md`
- `docs/03-多Agent智能助手 页面与输出实现规范.md`
- `docs/04-Chatflow Prompt 强约束清单.md`
- `docs/05-多Agent智能助手 判断过程事件规范.md`
- `docs/06-多Agent智能助手 Chatflow 与前端协议规范.md`
- `docs/07-业务场景与演示脚本.md`
- `docs/08-CRM 工具能力与数据协议.md`
- `docs/09-Agent 与 Chatflow 设计.md`
- `docs/10-实施路线与验收清单.md`
- `docs/ops-运行与配置手册.md`
- `docs/archive/`

阶段重点：

- PRD 是产品主干，必须存在。
- `00-文档索引.md` 负责告诉后续 Agent 哪些文档是当前依据。
- 历史文档只归档，不作为默认规划依据。

### 阶段 2：Agent 能力契约与验收确认

目标：固定 P0/P1 能力边界、自由问法验收和安全边界，避免把产品收窄成固定演示链路。

当前阶段 2 结论已沉淀到 `docs/10-实施路线与验收清单.md`。P0 不再按固定完整闭环验收，而按 Agent 能力契约验收：

```text
用户自由提问
  -> 主 Agent 判断任务意图、业务对象、信息缺口和风险复杂度
  -> 按需选择轻量回答、CRM 查询、销售运营、专项风险、多 Agent 评审、内容生成或写回确认
  -> Agent 根据任务输出合适形态的业务回答
  -> 只有出现显式写回意图时，才生成可复制草稿；真实 CRM 写回留到后续阶段
```

阶段重点：

- 明确 Agent 能力边界、自由问法验收和异常边界。
- 明确当前 V3 是只读演示链路，写回 CRM 是后续受控能力，不是默认结尾。
- 明确多 Agent 评审是高复杂度能力，不是所有商机问题的默认路径。

### 阶段 3：CRM 工具能力接入

目标：把已有 CRM 能力整理成 多Agent智能助手 可以稳定调用的工具。

最小工具集：

- 查询客户。
- 查询商机。
- 查询联系人。
- 查询跟进记录。
- 查询跟进计划。
- 按需生成跟进记录或跟进计划待写入内容。
- 当前只生成可复制写回草稿；真实写入 CRM 留到后续阶段。

阶段重点：

- 先做 多Agent智能助手 需要的薄封装，不重做 CordysCRM。
- 工具名称、输入参数、返回字段要服务 Chatflow 和前端展示。
- 写回动作必须保留人工确认。

### 阶段 4：Chatflow 多 Agent 改造与验证

目标：让多 Agent 价值真正体现出来，而不是只做结构上的分工。

阶段重点：

- 主 Agent 负责识别任务和分发。
- 销售 Agent 关注商机推进、客户关系和下一步动作。
- 财务 Agent 关注金额、回款、折扣和商业风险。
- 交付 Agent 关注资源、周期、范围和履约风险。
- 合同 Agent 关注条款、义务、审批和合规风险。
- 总结 Agent 按需输出结论、风险、建议；用户明确要求保存或写入时，当前只生成可复制草稿。

### 阶段 5：前端联调与体验打磨

目标：让 多Agent智能助手 在 CRM 内形成可演示的真实体验。

阶段重点：

- 双栏工作台承载当前会话、AI 判断过程、多结果文本澄清和写回确认。
- 对话区展示 AI 过程和最终结果，但不暴露内部技术标签。
- 当前写回类诉求在对话主工作区内生成草稿；真实写回确认、成功反馈或失败处理留到后续独立阶段。
- 页面文案使用销售用户能理解的业务语言。

### 阶段 6：演示验收与作品集包装

目标：把项目打磨成可以对外展示的作品。

阶段重点：

- 准备一条稳定演示数据。
- 准备 README 或作品集说明。
- 准备演示脚本和验收清单。
- 明确项目亮点：CRM 数据接入、Planner 证据调度、多 Agent 分工、业务建议与草稿生成。

## 5. 下一步动作

1. 维护 `docs/00-文档索引.md` 和 `docs/01-多Agent智能助手 PRD.md`，确保它们继续作为当前文档入口和产品主干。
2. 新建 `docs/10-实施路线与验收清单.md`，先明确 P0/P1 做到什么算完成。
3. 新建 `docs/08-CRM 工具能力与数据协议.md`，定义 P0 所需的最小 CRM 查询、写回、权限和人工确认规则。
4. 新建 `docs/09-Agent 与 Chatflow 设计.md`，在工具边界清楚后再设计多 Agent 分工和 Chatflow 调用链路。
5. 后续进入演示包装时，新建 `docs/07-业务场景与演示脚本.md`。
6. 联调和交付方式稳定后，新建 `docs/ops-运行与配置手册.md`。

## 6. 工作原则

- 先厘清问题，再重构文档，再进入实现。
- 多Agent智能助手 是项目主角，CordysCRM 是业务底座。
- PRD 是产品主干文档，后续设计、Chatflow、工具接入和验收都应围绕 PRD 展开。
- 多 Agent 价值必须通过真实业务数据查询、证据汇总、分工分析和可执行建议体现。
- AI 负责生成判断和草稿，不直接替用户审批或写入关键业务数据。
- 原型和演示页面只展示真实用户会看到的业务内容，不放内部术语、技术标签或设计说明。
- 旧文档在被新索引确认前，只作为参考材料，不作为当前规划依据。
