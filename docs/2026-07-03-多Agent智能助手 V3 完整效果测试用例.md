# 多Agent智能助手 V3 完整效果测试用例

> 本文档用于 多Agent智能助手 V3 发布后的人工集成测试和演示效果验收。测试用例根据当前前端代码与 `多Agent智能助手 - V3 Stable Enhanced.yml` 动态整理，重点覆盖对话体验、Markdown/图表渲染、图片输入、CRM 查询、风险分析、联网搜索、写回确认、多轮上下文和异常边界。

## 测试前置

- 前端地址：http://localhost:5173/#/agent/deal-desk
- 登录地址：http://localhost:5173/#/login
- 本地账号：`admin / CordysCRM`
- 后端健康检查：http://localhost:8081/is-login
- Dify Cloud 已导入当前 `多Agent智能助手 - V3 Stable Enhanced.yml`
- 导入后 Dify 应用名应保持 `多Agent智能助手 - V3 Stable Enhanced`，不要误显示为 V4 或其它版本名。
- 如测试真实 Dify Cloud 调用 CRM 工具，需保持固定 ngrok 域名可用：`https://your-domain.example.com/anonymous/ai/deal-desk/dify-tools`
- 建议准备 1 个本地 CRM 中真实存在的客户和 1 个真实存在的商机。本文默认示例为：
  - 客户：`南京数据治理中心`
  - 商机：`南京数据治理中心低优先级小单1期`
  - 如本地数据不存在，可替换为当前 CRM 中能搜索到的客户/商机名称。

## 分析依据

### 前端代码能力

- `index.vue`：承载对话页、会话列表、CRM 对象抽屉、图片预览弹窗。
- `useAiDealDeskChat.ts`：管理会话、发送、停止生成、@ 提及、图片附件、流式过程事件、待确认写回状态。
- `dealDeskDifyProvider.ts`：上传图片到 Dify，向后端传 `query / boundObject / activeWriteback / files`。
- `dealDeskChatflowAdapter.ts`：把 Chatflow 协议转成前端回复类型；只有 `text_analysis / writeback_confirm / writeback_result` 展示判断过程。
- `AssistantMarkdownPreview.vue` 与 `dealDeskMarkdown.ts`：支持 Markdown、表格、代码块复制、安全链接、HTML 清洗和流式 Markdown 修复。
- `ChartFence.vue` 与 `chartFencePlugin.ts`：支持 `chart` 代码围栏渲染柱状图、折线图、饼图、环图、漏斗图。
- `ComposerPanel.vue`：支持图片上传、输入框粘贴图片、最多 5 张图片、缩略图预览和移除。
- `ConversationTurn.vue` 与 `ProcessTimeline.vue`：展示用户引用、图片附件、判断过程折叠条和业务化过程事件。

### Chatflow V3 能力

- 用户输入支持 `uploaded_files` 文件列表，当前配置只支持图片。
- 图片链路：附件图片状态准备 -> 图片列表过滤 -> 图片/截图识别 -> 图片摘要整理。
- 主 Agent 参数抽取：识别 CRM 查询、商机总结、沟通生成、跟进计划、财务/交付/合同专项、综合评审、统计、写回、确认/取消/修改。
- CRM 工具链：根据路由规划是否调用 CordysCRM Tool API，再把结果标准化给后续 Agent。
- 外部情报链：CRM 上下文 -> 是否需要联网 -> 外部情报检索 Agent(FunctionCalling) -> 外部情报摘要。Agent 内部挂载 `current_time / Tavily Search / Tavily Extract`，由 Agent 自主决定是否搜索、是否抓取网页正文。简单问答、统计、对象选择、纯写回确认应跳过或不调用工具。
- 知识库链：复杂评审和专项风险进入规则查询改写、规则召回、重排和分支路由。
- Agent 分支：
  - 外部情报检索 Agent：只负责公开资料佐证，不做最终回答，不写回 CRM。
  - 快速问答 Agent：普通问题和图片只读回答。
  - 销售运营 Agent：CRM 查询、商机总结、沟通草稿、跟进计划、写回确认、统计解释。
  - 财务专项 Agent：折扣、付款、账期、回款风险。
  - 交付专项 Agent：上线周期、范围、定制、资源、验收风险。
  - 合同专项 Agent：验收、赔付、责任边界、数据安全和特殊条款。
  - 综合评审：销售、财务、交付、合同四个专家 Agent 后由协调总结 Agent 汇总。
- 写回链路：生成待确认草稿，保存会话变量；用户下一轮确认/取消/修改后进入写回结果。

## 通用验收口径

- 用户可见区域不应出现 Dify 节点名、内部路由、工具 payload、协议 JSON、原始 API 响应、token/耗时、`<think>` 或内部推理。
- 简单问题应短答，不应强制输出复杂报告、固定大纲或判断过程。
- 需要业务判断、写回、专项风险或综合评审时，应展示业务化判断过程；完成后默认收起，展开后步骤可读。
- CRM 内部事实、模型推断、外部公开资料应边界清楚。外部资料不能替代 CRM 数据、合同/报价原文或公司规则。
- 写回 CRM 必须先让用户确认；普通问答和分析不得自动写回。
- 联网搜索属于内部工具行为，用户可见回答只应展示必要来源链接和业务含义，不应暴露 Agent 工具选择、Tavily payload、节点名或搜索日志。
- 图片上传只验收本地图片上传和粘贴图片；当前版本不验收 URL 图片、PDF、Word、Excel、音频或视频解析。
- `实际运行结果` 与 `问题记录` 留空，测试时直接在对应位置粘贴截图、文字和问题描述。
