## 1. 修订目的

本文档用于指导 `多Agent智能助手` 工作台的页面设计稿、Figma 原型和前端骨架开发。它不是完整商机评审页 PRD，而是当前阶段工作台的实现型规范。

本版基于最新产品判断重写：

- 工作台是 AI 业务对话入口，不承担完整结构化商机评审页职责。
- 多 Agent 能力仍然保留，但在工作台中以文字结论、轻量判断过程和必要动作入口呈现。
- 前端只对真正需要用户选择的内容做富交互：客户/商机选择。
- P0 写入不做草稿确认卡，改为对话式确认：用户明确要求写入时，AI 用结构化文字展示将写入内容，用户回复确认后再执行写入。
- 其他 AI 输出统一按普通文本或 Markdown 渲染，避免为了展示多 Agent 把页面做成复杂报告页、表单页或状态面板。
- P0 不再设置固定第三栏，工作台先按"左侧历史 + 中间对话"的智能助手体验落地；业务对象通过输入框 `@` chip、路由对象和会话记忆承载，不在 AI 回复上方额外展示对象提示行。多结果通过 Markdown 文本澄清，用户输入完整客户名或商机名继续。
- 文件材料通过输入区的本地文件上传按钮引入，不使用 `@文件` 形式。上传文件只作为本轮回答的补充上下文，不代表 CRM 业务对象。
- 完整商机评审结构化页面后续可从商机列表或商机详情页发起，不作为工作台 P0 主链路。

当前阶段约束：

- 先做工作台页面状态和核心交互，再接入真实 Dify API 与 CRM 写入接口。
- 写入按真实业务动作设计，P0 走对话式确认；用户确认后必须调用现有 CRM 写入接口创建跟进记录或跟进计划。
- 新增一级菜单即可，不先改商机列表操作列和商机详情页入口。
- 优先复用 CordysCRM 现有页面容器、卡片、抽屉、按钮和表单风格。

## 2. 当前版本定义

下文中的 `P0` 指当前准备落地的工作台版本。P0 的目标是做出一个“能问、能查、能触发 Agent、能给深度 Markdown 业务判断、能在用户明确要求时确认后写入”的 多Agent智能助手 工作台。

P0 主目标：

- 用户能从一级菜单进入 多Agent智能助手 工作台。
- 用户能直接问普通规则问题，不需要先选择客户或商机。
- 用户能在输入框中通过 `@客户` 或 `@商机` 主动绑定本轮业务对象。
- 当用户问题需要业务对象且存在多个候选时，页面展示 Markdown 文本列表，不展示客户/商机候选选择卡。
- 当用户明确围绕某个商机提问时，系统能使用当前商机作为会话记忆。
- 当用户发起专项分析或多 Agent 商机评审时，系统能展示业务化判断过程和有证据边界的 Markdown 长文本结论。
- 当用户明确要求保存跟进记录或生成跟进计划时，AI 用结构化文字展示将写入内容，用户回复确认后真实写入 CRM。

P0 不做：

- 完整结构化商机评审详情页。
- 多 Agent 报告式结果页面。
- 多商机经营盘点或管理驾驶舱。
- 报价、审批、合同等高风险真实写回。
- 为每类 Agent 单独设计复杂结果卡。
- 写入草稿确认卡、草稿编辑抽屉或复杂写入表单。

## 3. 页面定位

多Agent智能助手 工作台应被设计成“带 CRM 上下文的 AI 业务对话台”，而不是完整业务报告页。

页面需要完成四类表达：

- 对话：用户直接输入问题、追问、补充条件。
- 记忆：系统在会话中保留当前客户、当前商机、上一轮关键条件和待确认写入内容，但页面不做固定上下文侧栏。
- 过程：业务任务展示可审计的判断过程，不展示模型原始思维链。
- 动作：需要用户补充业务对象时用文本说明；需要写入时通过对话完成确认。

工作台和完整评审页的职责边界：

| 形态 | 适用场景 | 输出方式 |
| --- | --- | --- |
| 多Agent智能助手 工作台 | 普通问答、规则咨询、对象查询、专项分析、多 Agent 商机评审、用户明确要求的写入 | 对话文本/Markdown + 轻量判断过程 + 会话记忆 + 对话式写入确认 |
| 完整商机评审页 | 从商机列表或商机详情发起正式评审，需要留痕和完整报告 | 固定结构页面 + 多 Agent 分项结果 + 风险汇总 + 写回确认 |

工作台可以发起商机评审，但输出应是深度 Markdown 业务分析。正文应至少区分 CRM 已确认事实、核心风险、缺失信息和下一步动作。例如：

```text
# 多Agent智能助手 业务判断

当前商机可以继续推进，但需要先补齐付款条件、验收标准和上线范围等关键字段。已确认的信息只能支持初步风险判断；未在 CRM 中出现的首付比例、赔付条款或上线周期不得被当作既成事实。
```

如后续需要固定结构报告、审批留痕或图表化展示，再进入独立完整评审页；当前工作台先以 Markdown 长文本承载高价值判断。

## 4. 主内容区展示规则

工作台主内容区按“会话回合”展示。每个回合对应用户一次输入和系统围绕该输入产生的回复、过程、Markdown 多结果澄清或对话式写入确认。

推荐回合结构：

1. 用户问题。
2. AI 文本回答或业务结论。
3. 判断过程，仅业务任务展示。
4. Markdown 多结果列表，仅对象不唯一时展示。
5. 写入确认消息，仅用户明确要求写入时展示。

不再为“继续追问”设计按钮，也不在 AI 回复下方展示追问引导。用户继续发送消息就是追问。空会话初始态可以保留少量示例问题，但会话开始后不再追加快捷追问。

P0 页面按钮只保留不改变对话状态的轻量动作，例如：

- `查看商机详情`
写入确认不作为 P0 页面按钮。用户通过继续发送消息完成确认，例如回复 `确认`、`修改一下内容` 或 `取消`。

## 5. 结构化判断过程

“判断过程”的节点定义、命名口径、展示顺序和场景模板，统一以 `docs/05-多Agent智能助手 判断过程事件规范.md` 为准。本文只保留页面实现必须遵守的展示边界。

页面级规则：

- 判断过程是用户可见的简洁业务过程，不是模型思维链、Prompt 或工具日志。
- 普通问答默认不展示判断过程；业务查询、专项分析、多 Agent 评审、写入确认才展示。
- 单次展示建议控制在 3-6 条节点，避免把工作台做成执行日志页。
- 节点文案必须使用业务语言，不直接暴露 Agent、workflow、tool_call 等内部概念。

交互规则：

- 业务任务执行过程中，判断过程默认展开，逐步更新当前进度。
- 业务任务执行完成后，判断过程默认收起为摘要条，例如"已完成 5 项检查：任务识别、商机资料、付款条件、风险判断、建议生成"。
- 用户点击摘要条后可展开查看每一步业务事件。
- 普通问答不展示判断过程，也不展示收起摘要。

## 6. 必要富交互块

工作台 P0 不再需要识别客户/商机候选卡富交互块。多结果、对象不唯一、对象不存在等场景统一由 AI 的 Markdown 文本回答承担。

本轮业务对象提示行暂不作为 P0 能力。写入确认、写入成功、写入失败都不做特殊前端卡片或状态样式，走对话式文字输出。

### 6.1 多结果 Markdown 澄清

当 AI 判断用户提到了某个客户或商机，但命中多个候选时，页面按普通 Markdown 渲染候选列表，不能让模型自行猜测，也不展示候选选择卡。

展示文案示例：

```markdown
我查到多个匹配商机，请输入完整商机名继续：

- **华东智造集团 - AI 客服升级项目**：阶段 方案评审，金额 880000，负责人 周雨晴
- **华东智造集团 - 数据中台二期**：阶段 商务沟通，金额 520000，负责人 张伟
```

展示规则：

- 候选必须来自真实 CRM 查询结果。
- 前端不从 Markdown 中解析候选对象，也不提供 `选择此商机` 续跑动作。
- 用户下一轮输入完整客户名或商机名后，系统重新搜索；唯一命中后写入当前会话记忆并继续回答。
- 表格只在列表、对比或行动计划确实更清晰时使用，不作为所有回答的默认格式。

### 6.2 对话式写入确认

当用户明确提出“保存为跟进记录”“写入 CRM”“生成跟进计划”等写入意图时，AI 才进入写入确认流程。正常问答、风险分析和多 Agent 评审不主动弹出写入建议，也不主动询问是否写入。

P0 不做写入草稿确认卡、不做编辑抽屉、不做复杂表单。AI 必须先用普通文本展示即将写入的内容，再等待用户回复确认；样式上参考常规 AI 工具的对话输出，不把待写入内容渲染成代码块、表格表单或状态卡。

P0 优先支持两类写入：

| 写入类型 | 目标对象 | 示例 |
| --- | --- | --- |
| 商机跟进记录 | 商机/客户 | “已完成商机评审，总体风险中高；下一步需确认付款节点、验收口径与交付范围。” |
| 商机跟进计划 | 商机/客户 | “与客户确认上线范围冻结、首付款比例、验收标准及延期责任。” |

确认前必须展示完整写入信息：

- 客户。
- 商机。
- 负责人。
- 联系人。
- 类型。
- 跟进方式或计划方式。
- 跟进时间或计划时间。
- 正文内容。

推荐话术：

```text
即将写入商机跟进记录：

客户：上海智联科技股份有限公司
商机：智联云平台二期项目
跟进方式：电话
负责人：张伟
内容：已完成风险评审，当前主要风险集中在首付款比例、上线范围和验收口径。下一步建议与客户确认付款节点和一期上线边界。

回复“确认”后写入 CRM；如果需要调整，请直接告诉我要修改的内容。
```

用户回复 `确认` 后，系统调用现有 CRM 写入接口创建跟进记录或跟进计划。用户回复修改意见时，AI 重新整理待写入内容并再次请求确认。用户回复取消时，本轮写入终止。写入成功或失败后，AI 直接用普通文本说明结果、原因和下一步，不使用成功/失败 banner 或字段表格。

### 6.3 后续优化项：本轮业务对象提示行

当前 P0 不在 AI 回复上方展示“本轮商机 / 本轮客户”提示行。原因是这类信息一旦出现，就必须处理长名称截断、无绑定对象空态、客户问题与商机问题切换、详情入口归属等边界；这些边界会分散当前主流程重点。

P0 的对象表达只保留三种方式：

- 用户主动在输入框中通过 `@客户` 或 `@商机` 绑定对象。
- 对象不唯一时，在中间对话流展示 Markdown 多结果列表。
- 选择或绑定后，系统把对象写入当前会话记忆，AI 直接基于该对象回答。

如果后续重新加入对象提示行，应采用紧凑 chip 或两行摘要，长名称需要省略号截断，并且只展示“查看详情”入口，不承载最近跟进、跟进计划等容易产生归属歧义的信息。

### 6.4 输入框 `@客户` / `@商机` 与文件上传

`@客户` / `@商机` 是 P0 的轻量亮点功能，用于让用户在提问前主动绑定本轮业务对象。它解决的是“这次问题到底围绕谁回答”的问题，比固定第三栏更清晰。

文件上传用于把用户本地的材料作为本轮问题的补充上下文，例如付款方案、合同条款、会议纪要或客户邮件。文件不代表 CRM 业务对象，只代表本轮回答可参考的材料。

`@客户` / `@商机` 交互：

- 用户在输入框输入 `@` 后，弹出客户/商机选择器。
- 选择器支持按客户名、商机名关键词搜索。
- 用户选择后，输入框中展示为对象 chip，例如 `@智联云平台二期项目`。
- 消息发送后，`@客户` / `@商机` 应体现在用户消息气泡中，使用小号引用 chip，不抢占正文注意力。
- `@客户` / `@商机` 提供“查看详情”入口，点击后跳转到 CRM 已有的客户或商机详情页。

文件上传交互：

- 用户点击输入区的上传按钮，选择本地文件上传。
- 上传完成后，输入框中展示文件 chip，例如 `📎 付款方案v2.pdf`。
- 消息发送后，文件 chip 体现在用户消息气泡中，提供“查看文件”入口。

通用规则：

- 发送消息时，前端同时提交原始文本、结构化对象绑定信息和已上传文件列表。
- P0 不额外展示本轮业务对象提示行；对象绑定通过输入框 chip 和会话记忆体现，AI 直接基于该对象回答。

P0 范围：

- 一次提问只支持绑定一个主对象。
- 主对象类型只支持 `customer` 和 `opportunity`。
- 文件上传可以和一个主对象同时存在，但文件只作为补充材料，不改变主对象归属。
- 用户已通过 `@` 绑定对象时，以 `@` 对象为本轮优先对象。
- 如果用户没有使用 `@`，仍然走 AI 自动识别；多结果用 Markdown 文本澄清。
- 如果 `@` 对象和文本描述明显冲突，AI 应提示用户确认，不直接写入或给高置信结论。

暂不做：

- 多个客户或多个商机同时绑定。
- 客户与商机的复杂组合分析。
- `@联系人`、`@合同`、`@订单` 等更多 CRM 对象类型。
- 跨会话永久 pin 对象。
- 在输入框内展示完整 CRM 字段。
- `@文件` 形式的文件引用（统一走上传按钮）。

## 7. 页面整体布局建议

视觉上推荐采用“CRM 外壳 + 左侧历史 + 中间会话流”的结构，把工作台先做成一个稳定的智能助手。

| 区域 | 建议宽度 | 职责 |
| --- | --- | --- |
| 全局侧边栏 | 复用系统布局 | 显示 CordysCRM 一级菜单，当前高亮 `多Agent智能助手` |
| 左栏：历史会话 | 240-280px，可收起 | 历史会话列表、搜索、时间分组、当前会话高亮 |
| 中栏：当前会话流 | 自适应 | 用户问题、AI 文本回答、判断过程、多结果澄清、对话式写入确认、输入区 |

布局原则：

- 输入区固定在中栏底部。
- 中栏按会话回合追加内容。
- 普通 AI 输出按文本或 Markdown 展示，正常渲染段落、列表和加粗，不渲染成代码块。
- 判断过程是轻量时间线，不做复杂 Agent 面板。
- 多结果澄清是主内容流中的普通 Markdown 文本，不放到右下角弱提醒；写入确认、写入成功、写入失败都以对话文本出现在主内容流中。
- 当本轮回答绑定了客户或商机时，P0 只通过输入框 chip、路由对象和会话记忆表达，不设置固定第三栏，也不额外展示对象提示行。

不建议的布局：

- 不建议把所有 AI 输出都转成卡片。
- 不建议为每个 Agent 做独立复杂结果区。
- 不建议在工作台里实现完整商机评审报告页。
- 不建议做“继续追问”按钮。

## 8. 页面状态模型

页面级状态：

| 状态 | 触发条件 | 页面表现 |
| --- | --- | --- |
| 有历史会话 | 用户点击一级菜单且存在历史 | 左侧会话列表可见，默认选中最近会话 |
| 无历史会话 | 首次进入或历史为空 | 中栏显示欢迎态和推荐问题 |
| 有业务对象 | URL query、输入框 `@` 或用户选择了客户/商机 | 写入当前会话记忆，AI 直接基于该对象回答；P0 不额外展示对象提示行 |
| 无业务对象 | 一级菜单直接进入且未选择对象 | 仍允许普通问答 |

回合级状态：

| 状态 | 说明 | 页面表现 |
| --- | --- | --- |
| `quick_answer` | 普通问答或规则解释 | 用户问题 + 文本回答 |
| `object_select` | 废弃兼容 | P0 降级为 `text_analysis` 文本展示 |
| `text_analysis` | 专项分析或多 Agent 评审 | 用户问题 + 判断过程 + Markdown 文本结论 |
| `writeback_confirm` | 用户明确要求写入，等待确认 | 展示待写入内容 + 等待用户回复确认 |
| `fallback` | 结构化解析失败 | 降级展示 AI 原始总结，不进入写入流程 |
| `failed` | 处理失败 | 保留用户问题，展示失败原因和重试入口 |

> [!NOTE]
> 回合状态 `failed` 表示整个回合处理失败，与第 5 节判断过程事件类型中的 `failed` 不同。事件级 `failed` 表示某一步骤失败（例如单个 Agent 调用失败），但回合整体可能仍有结论。只有当所有关键步骤都失败且无法给出任何结论时，回合状态才设为 `failed`。

## 9. Chatflow 输出到前端展示的映射

当前 Chatflow 可继续承担任务识别、参数抽取、Agent 路由和结果生成。前端不直接绑定 Dify 原始输出，而是通过 adapter 转成工作台展示模型。

> [!NOTE]
> `processEvents` 的业务含义、节点命名和场景组合方式，统一以 `docs/05-多Agent智能助手 判断过程事件规范.md` 为准。Chatflow 与前端 adapter 的详细字段协议，统一以 `docs/06-多Agent智能助手 Chatflow 与前端协议规范.md` 为准。本节保留为页面实现阶段的模型参考。

推荐展示模型：

```ts
type DealDeskTurnType =
  | 'quick_answer'
  | 'text_analysis'
  | 'writeback_confirm'
  | 'fallback'
  | 'failed';

type DealDeskRichBlockType = never;

interface DealDeskTurn {
  id: string;
  type: DealDeskTurnType;
  userText: string;
  boundObject?: DealDeskBoundObject;
  answerText: string;
  processEvents: DealDeskProcessEvent[];
  richBlocks: DealDeskRichBlock[];
  pendingWriteback?: DealDeskWriteConfirmPayload;
  actions: DealDeskAction[];
  attachments?: DealDeskFileAttachment[];
  createdAt: number;
}

interface DealDeskProcessEvent {
  id: string;
  type:
    | 'task_identified'
    | 'object_required'
    | 'object_selected'
    | 'context_loaded'
    | 'memory_used'
    | 'agent_planned'
    | 'agent_completed'
    | 'rule_matched'
    | 'risk_found'
    | 'draft_generated'
    | 'confirmation_required'
    | 'writeback_completed'
    | 'failed';
  status: 'running' | 'completed' | 'warning' | 'failed';
  title: string;
  summary: string;
  /** 规则编号、CRM 字段名或业务依据摘要，前端渲染为纯文本 */
  evidenceRefs?: string[];
}

interface DealDeskRichBlock {
  id: string;
  type: DealDeskRichBlockType;
  payload: unknown;
}

interface DealDeskAction {
  id: string;
  label: string;
  actionType:
    | 'select_object'
    | 'view_detail'
    | 'generate_follow_plan'
    | 'generate_follow_record'
    | 'open_full_review'; // P1+，P0 前端不渲染
  payload?: unknown;
}
```

本轮绑定对象：

```ts
interface DealDeskBoundObject {
  objectType: 'customer' | 'opportunity';
  objectId: string;
  objectName: string;
  customerName?: string;
  source: 'mention' | 'auto_detected' | 'selection' | 'route_query';
}
```

`route_query` 来源的生命周期规则：

- 从 URL 参数带入的客户或商机只作为会话初始记忆，不是整个会话的永久绑定。
- 用户后续通过 `@` 绑定了其他对象时，`@` 对象优先于 `route_query` 对象。
- 用户切换话题时，新回合按新话题重新判断对象，不强制沿用 `route_query` 初始对象。
- 如果用户没有主动切换且问题仍与初始对象相关，系统继续使用 `route_query` 对象。

多结果输出不再有专用 payload。Chatflow 将候选对象整理进 `answerText`，前端按 Markdown 渲染；对象真正唯一后再通过 `boundObject` 写入会话记忆。

`@` 对象搜索 payload：

```ts
interface DealDeskMentionSearchPayload {
  keyword: string;
  objectTypes: Array<'customer' | 'opportunity'>;
  selected?: DealDeskBoundObject;
}

interface DealDeskFileAttachment {
  id: string;
  fileName: string;
  fileSize: number;
  fileUrl: string;
  uploadedAt: number;
}
```

对话式写入确认 payload：

```ts
interface DealDeskWriteConfirmPayload {
  id: string;
  type: 'follow_record' | 'follow_plan';
  customerName: string;
  opportunityName?: string;
  ownerName?: string;
  contactName?: string;
  followType?: string;
  timeText?: string;
  content: string;
  status: 'awaiting_confirm' | 'confirmed' | 'cancelled' | 'failed';
}
```

跨回合写入确认状态流转规则：

- 当 AI 展示待写入内容时，本回合的 `pendingWriteback.status` 为 `awaiting_confirm`。
- 用户在下一回合回复“确认”后，前端应将上一回合的 `pendingWriteback.status` 更新为 `confirmed`，并在新回合展示写入结果。
- 用户回复修改意见时，AI 在新回合重新整理待写入内容，新回合的 `pendingWriteback` 为新的 `awaiting_confirm` 状态；上一回合的 `pendingWriteback.status` 保持 `awaiting_confirm` 不变（历史记录）。
- 用户回复“取消”时，上一回合的 `pendingWriteback.status` 更新为 `cancelled`。
- 写入失败时，`pendingWriteback.status` 更新为 `failed`，保留待写入内容，允许用户回复“重试”。
- 前端在渲染时，应遍历所有回合找到最后一个 `awaiting_confirm` 状态的 `pendingWriteback`，作为当前活跃的写入确认。

映射规则：

| Chatflow 结果 | 工作台展示 |
| --- | --- |
| `quick_answer` | 文本回答，不展示判断过程，不主动提示写入 |
| `sales_task` | 文本回答 + 轻量判断过程；只有用户明确要求写入时才进入对话式确认 |
| `finance_task` | 文本回答 + 财务判断过程，可提示风险和复核角色 |
| `delivery_task` | 文本回答 + 交付判断过程 |
| `legal_task` | 文本回答 + 合同风险判断过程 |
| `full_review` | 工作台中展示深度 Markdown 业务判断 + 判断过程，不展示结构化报告卡，不主动生成待写入内容 |
| 命中多个对象 | 展示 Markdown 多结果列表，等待用户输入完整名称 |
| 用户通过 `@` 绑定对象 | 直接作为本轮业务对象，优先于自动识别结果 |
| 用户明确要求写入 | 展示结构化待写入内容，等待用户回复确认 |

## 10. 演示脚本

P0 建议准备 5 条可稳定演示的业务脚本。

### 脚本 1：普通规则问答

用户输入：

```text
什么情况下商机需要进入 多Agent智能助手 评审？
```

页面表现：

- 直接输出规则解释文本。
- 不展示判断过程。
- 不展示候选选择。
- 不主动提示写入。

### 脚本 2：多个商机文本澄清

用户输入：

```text
帮我看一下智联这个项目的风险。
```

页面表现：

- AI 识别到多个相关商机。
- 主内容区展示 Markdown 商机列表。
- 用户输入完整商机名后，当前会话记忆更新，系统继续执行原任务或回答用户追问；P0 不额外展示对象提示行。
- 系统不展示候选卡片，不提供 `选择此商机` 按钮。

用户输入：

```text
这个付款方案风险大吗？
```

- 不主动生成待写入内容；如用户继续要求保存或生成计划，再进入对话式写入确认。

### 脚本 4：用户明确要求写入

用户输入：

```text
把刚才的评审结论保存为商机跟进记录，并生成下一步跟进计划。
```

页面表现：

- AI 用结构化文字展示即将写入的跟进记录和跟进计划。
- AI 提示用户回复 `确认` 后写入 CRM，或直接说明需要修改的内容。
- 用户回复 `确认` 后，系统真实写回 CRM，并在对话中反馈写回结果。
- 如果用户回复修改意见，AI 重新整理待写入内容并再次请求确认。

### 脚本 5：`@` 对象绑定后直接分析

- 展示轻量判断过程和文字结论。
- 不主动生成待写入内容。

## 11. CRM 上下文和写入接入点

P0 验收要求真实 CRM 查询能力可用，并要求显式写回场景能在用户确认后真实写入 CRM。前端联调前可以使用模拟数据验证页面形态，但不能把模拟写入作为写回能力的完成标准。

推荐路由：

- `/ai-deal-desk/index`
- `/ai-deal-desk/index?customerId=xxx&source=customer`
- `/ai-deal-desk/index?opportunityId=xxx&source=opportunity`

进入逻辑：

| 进入方式 | 页面行为 |
| --- | --- |
| 一级菜单直接进入 | 加载历史会话，允许普通问答 |
| 用户在会话中选择客户/商机 | 更新当前会话记忆，AI 直接基于该对象回答；P0 不额外展示对象提示行 |
| 带 `customerId` | 作为会话初始客户记忆，不展示固定右栏 |
| 带 `opportunityId` | 作为会话初始商机记忆，不展示固定右栏 |

写入接入原则：

- 正常问答和业务分析不主动发起写入确认。
- 用户明确要求写入后，才整理待写入内容。
- 写入前必须用结构化文字展示完整待写入内容。
- 用户回复 `确认` 后才调用写入接口。
- 用户回复修改意见时，以修改后的版本为准，并再次请求确认。
- 写入失败时保留本轮待写入内容，允许用户回复 `重试` 或 `取消`。
- 缺少唯一客户或商机 ID 时，不进入写入调用，并提示需要先选择客户或商机。

## 13. 页面骨架开发顺序

建议按以下顺序推进：

1. 新增一级菜单和空页面，确认 `/ai-deal-desk/index` 可访问。
2. 搭建 CRM 外壳、左侧历史和中间会话布局。
3. 接入历史会话或本地样例会话，默认选中最近会话。

5. 完成多结果 Markdown 澄清，验证多个商机列表和完整名称继续流程。
6. 完成轻量判断过程，验证业务任务可以展示事件流。
7. 完成对话式写入确认，验证展示待写入内容、用户回复确认、显式写回场景真实写回 CRM。
8. 补齐空状态、加载态、失败态。
9. 再考虑接 Dify Chatflow adapter。
10. 最后接 CRM 上下文和写入接口。

## 15. 本阶段验收标准

页面设计稿和前端骨架达到以下标准即可进入下一阶段：

- 一级菜单 `多Agent智能助手` 可见。
- 工作台主内容区像对话页面；业务对象通过 `@` chip、路由对象和会话记忆承载，P0 不额外展示对象提示行。
- 普通问题可以直接文本回答。
- 输入框支持 `@客户` / `@商机` 主动绑定本轮对象。
- 用户继续发送消息就是追问，页面不提供泛化的“继续追问”按钮。
- 当商机或客户不唯一时，页面展示 Markdown 多结果列表。
- 通过 `@` 或完整名称唯一命中客户/商机后，当前会话记忆更新，并自动继续原任务，不额外展示对象提示行。
- `@` 绑定对象优先于 AI 自动识别结果；冲突时要求用户确认。
- 业务任务可以展示轻量判断过程，但不展示模型原始思维链。
- 商机评审在工作台中以深度 Markdown 业务分析呈现，包含结论、事实依据、风险、信息缺口和下一步建议；当前阶段不新增结构化报告卡。
- 正常问答和业务分析不主动提示写入。
- 用户明确要求写入时，AI 用结构化文字展示待写入内容，并等待用户回复确认。
- 用户回复确认后才调用 CRM 写入接口。
- P0 不做写入草稿确认卡、草稿编辑抽屉或复杂写入表单。
- AI 输出优先按文本或 Markdown 展示，不做候选卡片、追问引导、成功失败卡片、写入表格或复杂前端交互。
- 页面视觉和交互与 CordysCRM 业务页保持一致。

## 16. 最终建议

多Agent智能助手 当前应先做成 CordysCRM 内的 AI 业务对话工作台。它不是完整商机评审报告页，也不是纯聊天页，而是一个能承载普通问答、业务对象选择、深度 Markdown 业务分析和对话式写入确认的入口。

完整商机评审结构化页面应作为后续独立业务动作，从商机列表或商机详情发起。工作台仍可触发同一套多 Agent 能力，但当前用 Markdown 长文本展示结论、证据、风险、缺口、行动建议和必要动作入口。
