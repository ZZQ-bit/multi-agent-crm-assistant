# 多Agent智能助手 验收方案与用例集

更新时间：2026-07-19

## 1. 验收目标

本轮验证 `多Agent智能助手` 是否达到“面试演示项目可用”水平。测试覆盖前端、Dify Chatflow、CRM Tool API、数据库事实和文档一致性，不套用生产系统的发布门槛。

本轮需要证明：

- 系统能够使用 CRM 中真实存在的客户和商机完成查询、总结、内容生成、计划生成、专项判断和复杂评审。
- 复杂评审能够按需综合销售、财务、交付和合同视角，不输出审批通过或确定性承诺。
- 当前 Chatflow 只生成草稿，不调用真实写回工具，也不声称已经保存。
- 页面能够展示流式回答、Markdown、历史会话和连续对话；失败时提供可理解的恢复动作。
- 测试问题不再使用系统中不存在的占位客户或商机。

## 3. 范围与边界

### 3.1 本轮范围

- CRM 客户、商机、联系人、跟进记录、跟进计划和报价读取。
- 对象精确解析、多候选返回、页面绑定对象和连续会话。
- 商机总结、沟通内容、跟进计划、财务/交付/合同专项判断。
- 销售、财务、交付、合同四视角复杂评审。
- 海尔智家、京东集团现有演示数据对应的外部情报场景。
- 前端 Markdown、历史会话、失败提示和重试能力。
- 只读边界、会话隔离、工具令牌和内部协议泄露检查。

### 3.2 不作为失败项

- 前端真实写回、生产级审批、完整审计和幂等恢复。
- 完整评审报告页、管理驾驶舱和生产级监控。
- 偶发模型措辞变化、轻微格式差异和不影响理解的冗余。
- 单次响应超过参考耗时，但仍能完成且不持续阻塞演示。

## 4. 测试数据约束

- 每条自然语言用例必须使用前置快照中真实存在的客户或商机。
- 禁止使用“华为、腾讯、测试客户A、CRM采购商机A”等占位对象。
- 多候选使用真实数据，例如南京数据治理中心关联的两个商机。
- 对象不存在只通过 Tool API 无效 ID 或参数校验验证，不虚构业务客户名称。
- 测试结束后再次导出数据库快照，比对状态签名和跟进数据，确认没有意外写回。

## 5. 质量评分与通过标准

每条回答按 100 分初评：任务完成度 25 分、CRM 核心事实 25 分、业务分析与建议 20 分、表达与展示 20 分、安全和只读边界 10 分。

| 分数 | 结论 |
| --- | --- |
| 80-100 | 表现良好，可直接演示 |
| 60-79 | 基本可用，存在优化空间 |
| 40-59 | 部分可用，需要修复或规避 |
| 0-39 | 未完成任务或出现明显错误 |

整体用例通过率达到 80%、平均质量评分达到 65 分、12 条前端核心场景通过率达到 90%，即可判定演示版本基本可用。路由和工具名称只作辅助判断，最终回答正确时不因内部组合差异直接扣分。

以下问题阻止演示验收：系统无法使用、主演示链路普遍失败、发生真实误写、系统性编造 CRM 数据、泄露密钥或内部协议。单条事实偏差只判该用例失败并记录缺陷。

## 7. 执行与证据

1. 运行 Chatflow 静态校验、后端 `AiDealDesk*` 单测和本地/公网只读 Tool API 冒烟。
2. 通过 Dify API 顺序执行 40 条业务用例，保存原始回答、耗时和错误。
3. 通过真实 CRM 页面执行 12 条核心前端用例，覆盖查询、总结、专项、评审、外部情报、草稿、失败重试和连续会话。
4. 使用 `score-acceptance-results.ps1` 按宽松标准初评，再结合数据库事实和前端证据人工复核。
5. 对失败用例定向重跑；最后导出后置数据库快照并比对状态签名。

主要命令：

```powershell
python tests\ai-deal-desk\validate-v3-chatflow.py
.\CordysCRM\mvnw.cmd -pl backend/crm -Dtest=AiDealDesk* test
& .\CordysCRM\backend\crm\scripts\ai-deal-desk-dify-tools-smoke.ps1
& .\tests\ai-deal-desk\run-v3-tests.ps1 -TestCasesFile .\tests\ai-deal-desk\v3-test-cases.json
```
