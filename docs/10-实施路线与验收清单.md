# 多Agent智能助手 实施路线与验收清单

## 1. 文档目的

本文档定义 `多Agent智能助手` 当前能力范围、持续验收口径和后续维护顺序。

项目按 Agent 能力契约验收，不把固定演示链路写入产品逻辑。

核心原则：

- 当前 Chatflow 不执行 CRM 写回；写回工具仅存在于后端预留接口。
- 多 Agent 评审是高复杂度能力，不是所有问题的默认路径。
- 跟进记录和跟进计划草稿只在用户明确要求生成时出现。
- 输出结构由 Agent 根据任务决定，不使用统一固定模板。
- 验收关注自由问法下的能力覆盖、路由正确性和安全边界。

## 2. 当前与后续能力边界

| 阶段 | 目标 | 必须具备 | 不要求 |
| --- | --- | --- | --- |
| 当前版本 | CRM 内多 Agent 销售协同工作台 | 自主路由、对象解析、CRM 查询、总结、专项判断、多 Agent 评审、外部情报、内容草稿、会话记忆、流式回答 | CRM 真实写回、完整评审报告页、管理驾驶舱、高风险字段更新 |
| 后续扩展 | 受控业务动作和更多展示形态 | 写回确认、权限、幂等、审计、完整评审页、更多对象类型 | 自动审批、自动报价、自动合同结论 |

P0 完成标准不是“每次都跑完整流程”，而是用户以多种问法进入时，系统能选择合适能力并给出业务上可信、边界清晰的响应。

## 3. Agent 能力契约

| 能力 | 触发方式 | 期望行为 | 验收重点 |
| --- | --- | --- | --- |
| 轻量回答 | 非 CRM 或泛销售管理问题 | 直接回答，不声称查询 CRM | 不误走 CRM 查询或多 Agent |
| CRM 对象查询 | 用户询问客户、商机、联系人、跟进记录、跟进计划 | 查询真实 CRM 数据；多候选用 Markdown 列表澄清 | 多候选不猜测、不出候选卡 |
| 商机/客户总结 | 用户要求总结进展、风险、最近动态 | 汇总 CRM 上下文，输出事实、风险、缺口和建议 | 不默认写回 |
| 专项风险判断 | 用户聚焦付款、折扣、交付、合同、销售推进 | 调用对应专项能力并输出有依据的 Markdown 分析 | 不展开不必要视角 |
| 多 Agent 评审 | 用户提出复杂成交方案、多维风险或明确评审 | 销售、财务、交付、合同按需分析并协调总结为深度 Markdown 业务判断 | 不输出审批结论 |
| 内容生成 | 用户要求生成话术、纪要、跟进文本或计划 | 输出可编辑文本 | 不声称已保存 |
| 内容与计划草稿 | 用户明确要求生成跟进记录、跟进计划、沟通话术或摘要 | 输出可复制、可修改的 Markdown 草稿 | 不声称已保存，不调用写回工具 |

## 4. 自由问法验收集

以下验收问题用于验证 Agent 的自主路由和能力组合。实际测试可以替换客户或商机名称，但不能把测试样例写死进产品逻辑。

| 类型 | 示例问法 | 期望路由 | 通过标准 |
| --- | --- | --- | --- |
| 非 CRM 轻量问题 | `多Agent智能助手 是什么？` | 轻量回答 | 给出概念解释，不声称查 CRM |
| CRM 客户查询 | `帮我查一下华东智造集团` | CRM 查询 | 返回客户信息；多结果用 Markdown 列表提示输入完整名称 |
| CRM 商机查询 | `智造集团有哪些商机？` | CRM 查询 | 返回相关商机，不进入评审 |
| 商机总结 | `总结一下这个商机最近进展` | CRM 查询 + 总结 | 输出摘要、进展、风险或下一步，不默认写回 |
| 财务专项 | `这个付款周期 90 天风险高吗？` | 财务专项 | 聚焦付款、账期、回款风险 |
| 交付专项 | `两周上线还能承诺吗？` | 交付专项 | 聚焦周期、范围、资源和验收风险 |
| 合同专项 | `合同里延期赔付和验收后付款有什么风险？` | 合同专项 | 聚焦条款和复核建议 |
| 复杂评审 | `帮我评审这个商机，折扣、付款、交付和合同都看一下` | 多 Agent 评审 | 多视角分析并给综合建议 |
| 内容生成 | `帮我生成一段给客户确认付款节点的话术` | 内容生成 | 输出可直接编辑使用的文本 |
| 跟进草稿 | `根据刚才结论整理一条跟进记录` | 内容生成 | 输出可复制草稿，明确未写入 CRM |

## 5. 内容草稿与写回边界验收

内容草稿不是默认输出。只有满足以下条件时才生成：

- 用户明确表达生成跟进记录、创建跟进计划、整理摘要或沟通内容。
- 当前存在唯一客户或商机对象。
- 草稿内容可被用户理解和复核。

草稿必须展示：

- 目标对象。
- 内容类型。
- 草稿正文。
- 缺失信息或必要风险提示。

当前 Chatflow 不调用 `create-follow-record` 或 `create-follow-plan`。如果后续启用真实写回，必须重新验收对象唯一性、权限、人工确认、幂等、审计和失败恢复。

## 6. 异常与边界验收

| 场景 | 期望行为 |
| --- | --- |
| 对象不唯一 | 展示 Markdown 候选列表并要求输入完整客户名或商机名，不进入最终分析 |
| 对象不存在 | 说明未找到，并提示可补充关键词 |
| 信息不足 | 给初步判断时说明不确定性，必要时追问 |
| CRM 查询失败 | 区分读取失败和确实没有数据 |
| 部分 Agent 失败 | 输出已完成视角，并说明缺失视角影响 |
| 内容生成失败 | 保留用户问题和对象上下文，允许重新生成 |
| 非 CRM 问题 | 走轻量回答，不声称查询 CRM |
| 高风险动作 | 不输出审批通过、财务已通过、合同可签、保证上线等结论 |

## 6.1 长文本质量与证据边界验收

| 场景 | 期望行为 |
| --- | --- |
| 分析类回答过短 | 商机风险、专项判断和多 Agent 评审不得只返回几行摘要，应包含结论、已确认事实、核心风险、缺失信息、下一步动作和复核角色 |
| 缺少付款条件 | 只能说明付款条件、首付比例或账期待确认，不得断言首付款比例偏低 |
| 缺少上线周期或交付范围 | 只能说明周期、范围、定制或集成边界待确认，不得断言两周上线或交付范围失控 |
| 缺少验收或赔付条款 | 只能说明验收标准、赔付或责任边界待确认，不得断言已经存在处罚条款 |
| 知识库命中规则 | 规则只能解释风险判断，不得替代 CRM 事实或编造字段值 |
| 表格使用 | 只在列表、对比、行动计划确实更清晰时使用，不强制每次输出表格 |

## 8. CRM 工具验收标准

CRM 工具应满足：

- `docs/08-CRM 工具能力与数据协议.md` 已成为 CRM 工具协议主依据。
- CRM 工具清单、入参、出参、错误码和 Dify 调用方式与后端实现一致。
- CordysCRM 后端具备 多Agent智能助手 Tool API 薄适配层，且复用现有 CRM Service，不重写客户、商机、跟进业务逻辑。
- 后端提供对象解析、客户/商机查询、上下文、统计和写回工具。
- 当前 YML 只使用读取类能力，不调用写回工具。
- 对象不存在、对象不唯一、参数缺失、权限不足、CRM 读取失败和写回失败都有标准错误码。
- 后端单测需通过：`./mvnw.cmd -pl backend/crm -Dtest=AiDealDesk* test`。
- 本地后端重启后需通过 Tool API 冒烟：`powershell -ExecutionPolicy Bypass -File CordysCRM\backend\crm\scripts\ai-deal-desk-tools-smoke.ps1`；如需验证真实跟进记录和跟进计划写回，追加 `-CreateWriteback`。
- `docs/08` 与后端 Controller、Service 和 Dify 工具 Schema 保持一致。

## 9. Agent 与 Chatflow 验收标准

Agent 与 Chatflow 应满足：

- `docs/09-Agent 与 Chatflow 设计.md` 已成为 Agent 和 Chatflow 改造主依据。
- Planner 能区分轻量回答、CRM 查询、商机总结、专项分析、多 Agent 评审、外部情报和内容生成。
- CRM 查询通过阶段 3 Tool API 完成，Dify Chatflow 不直接调用 CordysCRM 原始页面接口。
- Planner 使用 `resolve-crm-object` 查证用户明确提到的新对象；上下文节点使用 `get-customer-context`、`get-opportunity-context`。
- 多候选对象必须映射为 `OBJECT_AMBIGUOUS` 和 Markdown 文本澄清，Agent 不允许猜测客户或商机。
- 用户输入完整客户名或商机名后，Chatflow 能唯一命中对象、写入会话记忆并继续回答。
- 商机上下文由主 Agent 或上下文准备节点统一获取，再分发给销售、财务、交付、合同等专项 Agent；专项 Agent 不各自重复查询 CRM。
- 多 Agent 评审只在复杂商机问题或用户明确要求评审时触发，且至少覆盖销售、财务、交付、合同四个视角和协调总结。
- 内容生成请求输出可复制草稿，不调用 `create-follow-record` 或 `create-follow-plan`。
- Chatflow 最终输出继续符合 `DealDeskChatflowPayload`，不向用户展示 Prompt、工具调用、节点名、内部路由或 JSON 调试内容。
- 运行时代码不再导入 `dealDeskMockProvider`、本地候选对象或本地写回成功兜底；Demo SQL 种子数据可以保留。
- Dify 超时、CRM Tool API 失败、对象不存在、写回未接通等场景必须返回失败态或清晰错误说明，不返回假业务成功。
- 多客户/多商机场景通过 `object_select` 和 `answerText` 返回 Markdown 列表，不输出前端候选卡片。
- 分析类回答应给出依据、风险、信息缺口和下一步建议；表格只在列表、对比或行动计划更清晰时使用。
- 会话记忆必须可用：`@` 绑定、路由带入对象、上一轮唯一命中对象和 Planner 最近 5 轮记忆能够支持连续对话。
- 验收需覆盖本文档第 4 节自由问法验收集，并额外验证对象不唯一、对象不存在、CRM 读取失败和模型失败等可恢复场景。

## 10. 文档一致性验收标准

文档维护应满足：

- PRD 不再把固定完整闭环写成 P0 唯一路径。
- 文档明确当前 Chatflow 不执行 CRM 写回。
- 文档明确多 Agent 评审是高复杂度能力，不是所有问题默认路径。
- 文档明确输出结构由 Agent 根据任务决定。
- 验收清单覆盖自由问法、能力路由、对象解析、内容草稿和异常边界。
- 文档中的节点、字段、路由、组件和工具名称与当前代码及 YML 一致。
