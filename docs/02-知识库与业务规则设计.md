## 1. 文档目的

本文档定义 多Agent智能助手 的知识库内容、业务规则结构、Dify 检索配置、维护方式和 Chatflow 接入方法。

当前 Dify Chatflow 已经改为 Planner、证据收集调度、证据台账和按需多 Agent 编排。知识库不再直接挂在固定任务分支后面，而是作为证据来源之一进入统一证据台账，再供销售、财务、交付、合同、经营分析等 Agent 使用。

## 2. 知识库选型结论

### 2.1 当前方案

当前运行方案：

```text
项目 Markdown 维护规则
  -> 合并为 AI Deal Desk 规则知识库 - Dify 单页版.md
  -> 上传并索引到 Dify 内部知识库
  -> Chatflow Knowledge Retrieval 节点检索
  -> 证据台账和领域 Agent 使用检索结果
```

同时在项目仓库中保留 Markdown 版本：

```text
knowledge-base/ai-deal-desk/*.md
```

该结构同时满足：

- 分章节文件便于维护和复核。
- 单页文件与当前 Dify 数据集保持一致。
- 项目仓库中有可交付、可审查、可复用的规则文档。

### 2.2 不建议现阶段使用外部知识库

Dify 外部知识库适合已有自建 RAG 系统或第三方知识服务的团队。它不是简单填写一个 Notion 或飞书地址，而是要求先构建一个符合 Dify 规范的检索 API 服务。

外部知识库的实际链路是：

```text
Dify Knowledge Retrieval
  -> Dify 调用外部知识库 API 的 /retrieval 接口
  -> 外部服务自行检索
  -> 返回 content、score、metadata
  -> Dify 将返回片段作为上下文传给 LLM
```

现阶段不推荐的原因：

| 原因 | 说明 |
| --- | --- |
| 需要自建检索 API | 需要额外开发服务，实现 `/retrieval` 接口 |
| 需要公网可访问 | 如果使用 Dify Cloud，Dify 必须能访问该 API |
| 需要自维护检索逻辑 | 分段、索引、召回、打分都要自己处理 |
| 增加运行依赖 | 当前系统使用 Dify 内部知识库，不引入独立检索服务 |

### 2.3 Notion、飞书、Dify 的分工

| 工具 | 推荐用途 | 当前阶段是否使用 |
| --- | --- | --- |
| Notion | 规则协作和评审 | 未接入运行链路 |
| Dify 内部知识库 | 负责分段、索引、检索和应用内引用 | 当前使用 |
| 飞书文档 | 规则协作和评审 | 未接入运行链路 |
| Dify 外部知识库 | 连接自建 RAG / 第三方知识服务 | 暂不使用 |
| 项目 Markdown | 当前规则事实来源和 Dify 上传内容 | 当前使用 |

如果后续坚持使用飞书作为主知识源，可以有两种方案：

1. 飞书文档 -> 飞书 CLI 导出 Markdown -> 上传 / 同步到 Dify 内部知识库。
2. 飞书文档 -> 自建检索服务 -> Dify 外部知识库。

第一种适合当前项目，第二种适合后续高级扩展。

## 3. 源文档拆分

项目按领域维护分章节 Markdown，Dify 运行时使用一个单页知识库。以下内容分区保留在单页文件中：

| 知识库名称 | 用途 | 对应 Agent |
| --- | --- | --- |
| 多Agent智能助手 - 折扣与付款规则 | 折扣授权、首付、账期、分期、回款风险 | 财务风控 Agent |
| 多Agent智能助手 - 交付与定制规则 | 上线周期、标准模块、定制范围、资源风险 | 交付可行性 Agent |
| 多Agent智能助手 - 合同与验收规则 | 验收标准、延期赔付、付款节点、责任边界 | 合同风险 Agent |
| 多Agent智能助手 - 销售推进规则 | 商机阶段、客户意向、跟进节奏、推进计划 | 销售评估 / 销售运营 Agent |
| 多Agent智能助手 - 输出模板与写回规范 | Brief、跟进记录、跟进计划、审批说明草稿 | 综合回答 |
| CordysCRM 对象与 L2C 链路 | 客户、商机、合同、回款和发票对象关系 | Planner / 经营分析 Agent |
| CordysCRM 风险信号参考 | CRM 数据中的业务风险信号 | 领域 Agent / 经营分析 Agent |

拆分原因：

- 不同 Agent 需要的知识不同。
- 检索结果更干净，减少无关片段干扰。
- 后续可以单独维护、测试和替换。
- 便于审查权限边界和领域分工。

## 4. 每条规则的标准结构

每条知识库规则建议使用固定结构。

```markdown
## 规则名称

规则编号：
适用场景：
触发条件：
风险等级：
判断依据：
处理建议：
需复核角色：
Agent 使用提示：
禁止事项：
示例输入：
期望输出要点：
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| 规则编号 | 用于引用和测试，例如 `FIN-001` |
| 适用场景 | 财务、交付、合同、销售或输出模板 |
| 触发条件 | 用户输入或 CRM 上下文满足什么条件 |
| 风险等级 | 低 / 中 / 高 |
| 判断依据 | 为什么这样判断 |
| 处理建议 | Agent 应给出什么业务建议 |
| 需复核角色 | 销售主管、财务、交付、法务等 |
| Agent 使用提示 | 提醒 Agent 如何引用该规则 |
| 禁止事项 | 防止越权表达 |
| 示例输入 | 用于测试召回 |
| 期望输出要点 | 用于验收 Agent 输出 |

## 5. 知识库内容规划

### 5.1 折扣与付款规则

目标：让财务风控 Agent 能判断折扣、付款、账期、分期和回款风险。

第一版规则：

| 编号 | 规则 | 风险等级 |
| --- | --- | --- |
| FIN-001 | 85 折以下需要销售主管复核 | 中 |
| FIN-002 | 80 折以下需要更高层级审批 | 高 |
| FIN-003 | 首付款低于 50% 标记为高风险 | 高 |
| FIN-004 | 账期超过 60 天需要财务复核 | 中 |
| FIN-005 | 账期超过 90 天默认高风险 | 高 |
| FIN-006 | 分三期及以上付款需要明确付款节点 | 中 |
| FIN-007 | 尾款绑定验收后支付需关注验收标准 | 中 |
| FIN-008 | 低折扣叠加长账期应升级为高风险 | 高 |

### 5.2 交付与定制规则

目标：让交付可行性 Agent 能判断上线周期、标准范围、定制需求和资源风险。

第一版规则：

| 编号 | 规则 | 风险等级 |
| --- | --- | --- |
| DEL-001 | 标准模块交付周期低于 2 周需谨慎承诺 | 中 |
| DEL-002 | 两周内全量上线默认高风险 | 高 |
| DEL-003 | 定制看板、报表、流程改造需确认范围 | 中 |
| DEL-004 | 非标集成需求需确认接口和联调资源 | 高 |
| DEL-005 | 未明确验收标准不得承诺固定上线日期 | 高 |
| DEL-006 | 定制需求建议拆分一期标准上线、二期定制 | 中 |
| DEL-007 | 客户强绑定上线日期需交付负责人复核 | 高 |
| DEL-008 | 资源未锁定时只能给出计划建议，不能给承诺 | 高 |

### 5.3 合同与验收规则

目标：让合同风险 Agent 能识别延期赔付、验收、责任边界和付款节点风险。

第一版规则：

| 编号 | 规则 | 风险等级 |
| --- | --- | --- |
| LEG-001 | 延期赔付条款默认高风险 | 高 |
| LEG-002 | 验收标准不清不得接受验收后付款 | 高 |
| LEG-003 | 客户单方验收权需法务复核 | 高 |
| LEG-004 | 定制成果归属不清需明确知识产权边界 | 中 |
| LEG-005 | 数据安全责任边界不清需法务复核 | 高 |
| LEG-006 | 无限责任或间接损失赔偿不得直接接受 | 高 |
| LEG-007 | 付款节点和验收节点绑定时需同步财务复核 | 中 |
| LEG-008 | 合同条款风险不能由 Agent 给最终法律意见 | 高 |

### 5.4 销售推进规则

目标：让销售评估 Agent 和销售运营 Agent 能生成商机总结、跟进计划和沟通建议。

| SAL-002 | 决策人缺失时建议补充关键联系人 | 中 |

## 7. Chatflow 当前接入链路

当前 V3 Stable Enhanced 接入方式：

```text
Planner 任务理解与规划
  -> 任务计划标准化
  -> CRM 与附件证据准备
  -> knowledge_policy_gate 读取 Planner 知识策略
  -> knowledge_query_decider 结合 CRM 与附件事实决定是否检索
  -> query_rewrite 生成检索问题
  -> knowledge_gate 执行检索门控
  -> Knowledge Retrieval: deal_rules_knowledge
  -> knowledge_evidence 整理命中规则
  -> evidence_ledger 汇总 CRM、知识库、外部情报、附件证据
  -> gap_check 判断证据是否足够
  -> agent_router 按需进入销售、财务、交付、合同、经营分析 Agent
  -> CRM 业务回答生成统一成文
  -> business_answer_stream 直接输出 Markdown
```

当前知识库节点的职责：

1. Planner 输出 `knowledge_policy` 和 `knowledge_goal`。
2. `knowledge_query_decider` 基于用户问题、CRM 和附件事实输出单条规则检索问题。
3. `query_rewrite` 校验查询并限制长度。
4. `deal_rules_knowledge` 最多返回 8 条结果。
5. `knowledge_evidence` 去重并整理规则证据。
6. `agent_router` 根据任务计划选择使用证据的领域 Agent。

这意味着知识库不是“财务分支专属”或“合同分支专属”的孤立节点，而是被统一沉淀进本轮证据台账。复杂商机评审时，销售、财务、交付、合同 Agent 可以共同消费同一份规则证据；单一专项问题也只启用必要 Agent。

规则使用边界：

- 知识库规则只能解释风险判断，不能替代 CRM 事实。
- CRM 没有确认的字段，不允许 Agent 用规则自行补全。
- 召回为空时，Agent 可以基于 CRM 事实给有限判断，但必须说明缺少规则依据。
- 规则命中不能输出“审批通过”“财务已通过”“合同可签”等越权结论。
- 输出模板类规则只约束正文结构，不改变当前 V3 只读、不写回 CRM 的边界。

## 8. 维护流程

当前维护流程：

```text
业务规则调整
  -> 修改分章节 Markdown
  -> 同步单页版 Markdown
  -> 在 Dify 中替换文档并重新索引
  -> 在知识库调试页测试召回
  -> 在 Chatflow 中测试典型问题
```

如果使用飞书：

```text
业务规则调整
  -> 修改飞书文档
  -> 使用飞书 CLI 导出 Markdown
  -> 上传 / 替换 Dify 内部知识库文档
  -> 同步项目 Markdown 备份
```

## 9. 验收标准

| 验收项 | 标准 |
| --- | --- |
| 内容覆盖 | 单页知识库覆盖财务、交付、合同、销售和 CordysCRM 对象规则 |
| 规则质量 | 每条规则包含触发条件、风险等级、处理建议、复核角色 |
| 召回效果 | 典型问题能召回对应规则 |
| Agent 输出 | 输出中能体现规则依据，并区分 CRM 已确认事实、规则推断和缺失信息 |
| 可维护性 | 规则可在 Notion 或 Markdown 中独立维护 |
| 边界控制 | Agent 不输出审批通过、法律结论、交付承诺等越权表达 |
| 接入链路 | 规则证据进入 evidence_ledger，并由按需 Agent 消费，不回退到旧的固定分支路由 |

## 10. 扩展边界

后续如果需要展示更高级能力，可以考虑：

- 建设外部知识库 API，连接飞书或自建 RAG。
- 为每条规则增加 metadata，例如适用行业、客户类型、产品线、风险等级。
- 将规则结果和 CRM 字段联动，例如商机金额、客户级别、历史回款表现。
- 将知识库召回结果在前端 Agent 过程区展示为“判断依据”。
