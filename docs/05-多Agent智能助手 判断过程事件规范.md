## 1. 文档目的

本文档用于固定 `多Agent智能助手` 工作台里“判断过程”的定义、展示边界、节点规则和场景模板。

这里的“判断过程”不是模型思维链，也不是 Agent 调试日志，而是用户可见的简洁业务过程。它的作用只有一个：让用户知道 AI 为了得出这次业务结论，完成了哪些关键业务检查。

## 2. 核心定义

判断过程应被定义为：

- AI 为完成当前业务任务所执行的关键业务检查。
- 面向用户可见、可理解、可复核的轻量事件流。
- 用业务语言表达，不用技术语言表达。
- 由 Dify SSE 节点状态提供实时信号，再通过统一映射规则转成标准业务事件。

判断过程的唯一展示链路是：

```text
Dify SSE 节点运行状态
-> 后端统一映射为标准过程事件类型
-> 前端按事件类型生成固定用户文案和摘要
```

Chatflow、后端、前端不能各自维护一套判断过程文案。Chatflow 可以负责业务判断和结构化结果，后端负责把实时节点状态转成标准事件，前端负责按标准事件展示。

判断过程不是：

- 模型原始思考过程。
- Prompt 内容。
- 工具请求体或返回体。
- Agent 调度面板。
- Token、模型、耗时等调试信息。
- 为了演示而硬凑的固定五步模板。

## 3. 展示原则

### 3.1 总原则

- 业务任务与普通规则问答都展示判断过程，但展示强度不同。
- 每次只展示和当前任务直接相关的节点，不补无效步骤。
- 节点数量由当前回合实际命中的标准事件决定，不再由前端按任务类型二次裁剪条数。
- 普通规则问答和业务分析共用同一套映射规则，不再额外维护“轻量三步过程”这类单独展示规则。
- 节点文案必须让销售、销售管理者或业务同事一眼看懂。
- 节点顺序遵循“先识别、再核对、后判断、最后产出”的自然业务顺序。
- 用户可见文案与事件类型保持固定映射，不根据 Chatflow 节点名称或模型输出临场变化。

### 3.2 展开与收起

- 执行过程中默认展开，逐步更新。
- 执行完成后默认收起为一条摘要。
- 用户点击摘要后可以展开查看完整步骤。
- 普通问答同样展示判断过程，但默认保持轻量摘要和轻量步骤，不展示复杂业务分析链路。

### 3.3 文案规则

- 优先使用“已识别”“已读取”“已核对”“已发现”“已生成”“等待确认”这类业务动作表达。
- 可以出现“客户”“商机”“付款条件”“合同条款”“交付范围”“跟进计划”等业务词。
- 不出现 `workflow`、`route`、`tool_call`、`agent_completed`、`memory_used` 这类内部表达。
- 不出现“模型认为”“系统推理中”“正在思考”这类空泛表述。

## 4. 节点模型

判断过程的底层结构保留事件类型，用户可见层按下表固定渲染。除 `status` 带来的“正在/已完成”差异外，同一事件类型不再维护多套文案。

| 事件类型 | 默认展示 | 何时出现 | 运行态文案 | 完成态/展示文案 |
| --- | --- | --- | --- | --- |
| `task_identified` | 是 | 已识别出当前任务类型 | 正在识别本轮任务 | 已识别本轮任务 |
| `object_required` | 是 | 缺少唯一客户或商机，需要用户选择 | 需要先确认本轮分析对象 | 需要先确认本轮分析对象 |
| `object_selected` | 视情况 | 用户通过 `@`、完整名称唯一命中或路由对象完成绑定，或本轮对象发生切换 | 正在确认本轮分析对象 | 已确定本轮分析对象 |
| `context_loaded` | 是 | 已读取 CRM 对象资料、跟进记录、计划、附件等 | 正在读取相关业务资料 | 已读取相关业务资料 |
| `memory_used` | 否 | 复用了当前会话已有条件 | 正在参考当前会话信息 | 已参考当前会话信息 |
| `rule_checked` | 是 | 核对付款、报价、交付、合同、阶段规则等 | 正在核对关键业务条件 | 已核对关键业务条件 |
| `risk_found` | 视情况 | 发现明确风险或信息缺口 | 正在识别风险与信息缺口 | 已识别风险与信息缺口 |
| `suggestion_generated` | 是 | 已形成结论或下一步建议 | 正在生成结论和下一步建议 | 已生成结论和下一步建议 |
| `confirmation_required` | 是 | 进入写入确认或下一步动作确认 | 等待你确认下一步操作 | 等待你确认下一步操作 |
| `writeback_completed` | 是 | 写回 CRM 成功 | 正在写入 CRM | 已完成 CRM 写入 |
| `failed` | 是 | 关键步骤失败且需要用户感知 | 暂时无法完成本轮处理 | 暂时无法完成本轮处理 |

说明：

- `memory_used` 默认作为内部事件保留，不进入用户界面；只有明确需要解释“沿用上一轮确认对象”时才可展示。
- `object_selected` 只有在对象切换、唯一命中或对象确认时展示；页面初始绑定对象时不必展示。
- `risk_found` 没有明确风险或信息缺口时不展示。
- `agent_planned`、`agent_completed` 这类内部步骤不进入用户界面；如确有业务价值，必须先映射为上表中的标准事件类型。
- `rule_checked`、`risk_found`、`suggestion_generated` 是当前阶段最核心的三类业务节点，应优先保留。

## 5. 推荐顺序

一个完整但克制的判断过程，推荐遵循下面的顺序：

1. 识别任务。
2. 确认对象。
3. 读取资料。
4. 核对条件。
5. 发现风险或确认机会点。
6. 生成建议或等待确认。

写入场景在最后追加：

7. 等待确认。
8. 写入结果。

不是每次都要走完全部步骤。没有对象歧义时，可以跳过“确认对象”；没有风险时，可以不单独展示“发现风险”，直接进入“生成建议”。

## 6. 任务类型与展示强度

| 任务类型 | 是否展示判断过程 | 展示建议 |
| --- | --- | --- |
| 普通规则问答 | 建议展示 | 展示本轮真实命中的标准事件，不再固定轻量条数 |
| 客户/商机查询 | 视情况展示 | 只展示本轮实际触发的对象确认、资料读取等事件 |
| 商机进展总结 | 建议展示 | 重点关注资料读取与结论生成，但不人为限制节点条数 |
| 付款/报价风险分析 | 建议展示 | 重点关注条件核对与风险识别，是否展示取决于真实节点映射 |
| 交付风险分析 | 建议展示 | 重点关注范围、周期、资源核对，是否展示取决于真实节点映射 |
| 合同风险分析 | 建议展示 | 重点关注条款核对与争议风险，是否展示取决于真实节点映射 |
| 复杂商机评审 | 必须展示 | 展示当前 Chatflow 实际命中的关键检查节点 |
| 写入 CRM | 必须展示 | 只展示对象确认、等待确认、写回结果等真实事件 |

## 7. 场景模板

### 7.1 客户或商机查询

适用场景：

- “帮我看一下智联这个项目是什么阶段”
- “这个客户最近谁在跟”

推荐事件：

- `object_selected`，仅用户刚完成对象确认或对象切换时展示。
- `context_loaded`
- `suggestion_generated`

### 7.2 商机进展总结

适用场景：

- “帮我总结当前商机进展”

推荐事件：

- `task_identified`
- `context_loaded`
- `suggestion_generated`

### 7.3 付款或报价风险分析

适用场景：

- “这个付款方案风险大吗”
- “当前报价有没有问题”

推荐事件：

- `task_identified`
- `context_loaded`
- `rule_checked`
- `risk_found`
- `suggestion_generated`

### 7.4 交付风险分析

适用场景：

- “这个项目两周上线靠谱吗”

推荐事件：

- `task_identified`
- `context_loaded`
- `rule_checked`
- `risk_found`
- `suggestion_generated`

### 7.5 合同风险分析

适用场景：

- “这个条款有没有风险”

推荐事件：

- `task_identified`
- `context_loaded`
- `rule_checked`
- `risk_found`
- `suggestion_generated`

### 7.6 复杂商机评审

适用场景：

- “请对这个商机做一次完整评审”

推荐事件：

- `task_identified`
- `context_loaded`
- `rule_checked`
- `risk_found`
- `suggestion_generated`

说明：

- 这里可以体现多视角检查，但不要把每个 Agent 单独做成一条技术节点。
- 用户可见层更关心“检查了哪些业务面”，而不是“调了哪几个 Agent”。

### 7.7 写入跟进记录或跟进计划

适用场景：

- “把刚才的结论保存为跟进记录”
- “生成下一步跟进计划”

推荐事件：

- `suggestion_generated`
- `object_selected`，仅写入对象刚被确认或切换时展示。
- `confirmation_required`

确认后：

- `writeback_completed`

## 8. 摘要条规则

执行完成后，判断过程折叠为一句摘要。摘要只做概括，不重复整段结论。

推荐格式：

```text
已完成 5 项检查：任务识别、资料读取、条件核对、风险判断、建议生成
```

可替换示例：

- 已完成 4 项检查：对象确认、资料读取、条件核对、建议生成
- 已完成 5 项检查：任务识别、资料读取、条件核对、风险判断、建议生成
- 已完成 3 项处理：建议生成、对象确认、等待确认

摘要条要求：

- 控制在一行内，超出时省略尾部描述。
- 使用名词化概括，不重复完整句子。
- 不出现技术术语。

## 9. 边界情况

### 9.1 缺少对象

如果任务必须依赖客户或商机，但当前没有唯一对象：

- 判断过程可以展示 `object_required`，用户可见文案固定为“需要先确认本轮分析对象”。
- 中间对话流同时展示 Markdown 多结果澄清。
- 不提前输出高置信结论。

### 9.2 多个候选对象

如果命中多个客户或商机：

- 判断过程只提示“需要先确认本轮分析对象”。
- 真正的对象确认由用户输入完整名称、`@` 绑定或路由对象承载。
- 用户选择后再继续原任务。

### 9.3 信息不足

如果资料不完整：

- 可以展示 `risk_found`，用户可见文案固定为“已识别风险与信息缺口”。
- 回答正文明确缺的是什么，例如付款节点、验收标准、最终采购方。
- 不因为信息不足而伪造低风险结论。

### 9.4 部分步骤失败

如果某个检查失败但仍可给出部分结论：

- 判断过程可展示 `failed`，用户可见文案固定为“暂时无法完成本轮处理”。
- 回答正文说明影响范围。
- 不把整个回合直接打成失败。

## 10. 前端、后端与 Chatflow 落地要求

- Chatflow 不负责拼装用户可见判断过程文案，只负责输出 `turnType`、`answerText`、`writeback`、`boundObject` 等业务结果；历史 `objectSelect` 仅为兼容字段，P0 不消费。
- Dify SSE 节点状态可以作为实时过程输入，但必须先由后端映射为本文档定义的标准事件类型。
- 后端是过程事件的唯一转换层，负责把节点开始、完成、失败映射成 `running`、`completed`、`warning`、`failed`。
- 前端只消费标准事件类型、状态和顺序，按第 4 节固定文案渲染，不从 Chatflow 正文或节点名称生成新文案。
- 前端没有收到标准 `processEvents` 时，不补业务过程文案；只展示正文或降级提示。
- 如果只是普通问答，前端也展示轻量判断过程，避免用户误以为当前功能未正常执行。
- 如果是业务任务，执行中展开、完成后折叠。
- Chatflow 与前端之间的字段协议，以 `docs/06-多Agent智能助手 Chatflow 与前端协议规范.md` 为准。

当前这版 `多Agent智能助手` 的分析阶段，判断过程只允许由后端显式登记的 Chatflow 节点标题触发，不允许按“节点职责关键词”模糊猜测。节点标题映射明细以后端 `AiDealDeskService` 中的映射常量和 `AiDealDeskStreamingEventParsingTest` 为准；本文档只维护用户可见业务语义，不长期双维护节点标题表。

补充规则：

- `suggestion_generated` 的完成态只能由最终输出节点触发，中间 `Agent` 节点成功结束时不展示“已生成结论和下一步建议”。
- 当前 Chatflow 没有独立的“风险识别完成”用户节点，因此分析阶段默认不从任意 `Agent` 或 `LLM` 节点推断 `risk_found`。
- `object_required`、`object_selected`、`confirmation_required`、`writeback_completed` 属于对象确认和写回交互阶段，不由分析阶段节点标题直接触发。
- 前端只做事件过滤、去重、固定顺序、状态优先级和固定文案渲染，不再按 `quick_answer`、`text_analysis` 等回合类型额外压缩显示条数。
- 后端与前端都不再为“过程看起来完整”而补假事件；没有真实事件时就不显示过程。
- 如果将来 Dify 里修改了节点标题，必须同步更新后端映射常量和对应测试；文档仅在业务语义或字段协议变化时更新。
- 当前 Chatflow 不需要再为判断过程新增额外参数；只要后端映射常量与当前导入的 Chatflow 节点保持同步，后端就能基于 SSE 做唯一映射。

## 11. 禁止项

以下内容不得作为判断过程直接展示给用户：

- “正在调用财务 Agent”
- “命中 route_finance_task”
- “memory_used = true”
- “tool_call success”
- “LLM reasoning”
- “Prompt 解析完成”
- “token 使用量”
- “耗时 2.4s”

这些信息如果确实需要保留，只能作为研发调试信息存在，不进入工作台用户界面。

## 12. 最终结论

`多Agent智能助手` 的判断过程，最终应固定为“用户可见的简洁业务过程”。

它既不能退化成纯黑盒回答，也不能膨胀成技术执行日志。最终规则是：节点状态提供实时性，后端负责标准化，前端负责固定文案展示。这样用户看到的过程稳定、一致、可解释，也不会因为 Chatflow 节点拆分或模型措辞变化而飘。
