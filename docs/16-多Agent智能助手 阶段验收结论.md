# 多Agent智能助手 阶段验收结论

更新时间：2026-07-19

## 1. 最终结论

本阶段结论：**演示可用**。

当前版本已经能够在桌面端完成“查询真实 CRM 上下文、生成业务判断或草稿、复杂商机多视角评审、连续追问”的完整演示。核心事实、只读边界和页面展示满足面试项目要求。

该结论仅表示适合当前桌面面试演示，不表示已经达到生产上线、真实写回或企业安全合规标准。

## 2. 结论依据

- 40 条真实数据 API 用例最终 40/40 有效返回，通过率 100%。
- 宽松自动质量评分平均 98.2 分，高于 65 分验收线。
- 12 条真实前端核心场景 12/12 通过，其中 1 条首次失败后重试成功。
- Chatflow 54 节点静态校验通过。
- 后端 69 个 `AiDealDesk*` 单测全部通过。
- 本地和公网 Tool API 冒烟全部通过。
- 数据库前后状态签名一致，没有意外写回。
- 未发现用户可见的内部协议、工具令牌或调试字段泄露。

## 3. 建议演示链路

面试时建议优先使用以下桌面端链路：

1. `看一下华东智造集团有哪些商机。`
2. `总结华东智造集团AI客服升级项目当前进展，重点说已确认事实、风险和下一步。`
3. `单独分析华东智造集团AI客服升级项目的折扣和付款风险。`
4. `完整评审华东智造集团AI客服升级项目，从销售、财务、交付、合同四个视角给出综合判断和下一步。`
5. `把华东智造集团这次折扣、付款和交付讨论整理成一条跟进记录草稿，不要保存。`
6. 在同一会话追问：`那付款风险呢？`

备用演示对象：

- 华北装备制造集团高风险大单1期：8 折、30% 首付、分期和赔付组合风险。
- 杭州客户成功咨询合同风险单1期：一期/二期交付拆分。
- 南京数据治理中心：真实多候选、失败状态和证据不足边界。
- 京东集团售后工单智能化项目：CRM 与公开情报结合。

## 4. 演示注意事项

- 只使用桌面端，不把移动端作为本轮成果展示。
- 演示前启动 MySQL、Redis、后端和 ngrok，并先执行一条短查询预热。
- 不同时运行批量测试和前端演示，避免 Dify Cloud 并发拥塞。
- 复杂评审通常需要 60-130 秒，应提前说明系统正在进行多视角判断，并准备短问题作为备用。
- 如页面出现请求失败，使用“重试”恢复；本轮已验证重试链路可用。
- 不使用“金额超过 100 万的商机明细”“列出全部商务采购商机”等全量筛选作为主演示问题，当前工具只能稳定返回汇总。
- 当前只生成草稿，不点击或宣称已经写回 CRM。
- 外部情报结果随时间和公开来源变化，不要求每次文字完全一致。

## 5. 遗留风险

当前主要遗留项为：Dify 长流程稳定性和耗时、全量筛选明细不足、移动端布局、内部调试请求的拒绝文案、traces 采集、演示 Tool Token 配置。

这些问题不阻塞桌面面试演示，但在后续继续完善时建议按以下顺序处理：

1. 优化复杂评审链路耗时和超时重试策略。
2. 补充按金额、阶段返回商机明细的读取工具。
3. 将 Tool Token 移出 YAML，改为 Dify 环境密钥并完成轮换。
4. 优化移动端导航和历史栏折叠。
5. 增加明确的内部调试信息拒绝回答。
6. 如果启用真实写回，再单独验收人工确认、权限、幂等、审计和失败恢复。
