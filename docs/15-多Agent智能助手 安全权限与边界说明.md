# 多Agent智能助手 安全权限与边界说明

更新时间：2026-07-19

## 1. 当前安全定位

当前版本是用于面试演示的 CRM 内多 Agent 销售协同工作台，不是生产审批系统。Chatflow 的职责是读取 CRM、生成判断和草稿；不自动审批、不自动报价，也不执行真实 CRM 写回。

本说明区分“已经验证的边界”和“未来上线前仍需补齐的能力”，避免把 Prompt 约束描述成完整安全体系。

## 2. 已验证能力

| 边界 | 实现与证据 | 本轮结果 |
| --- | --- | --- |
| Tool API 令牌 | 匿名 Tool API 要求 `X-DIFY-TOOL-TOKEN` | 错误令牌返回 `UNAUTHORIZED` |
| 工具执行身份 | Tool Controller 使用固定演示用户和组织执行，并清理线程上下文 | Controller 单测通过 |
| CRM 数据权限 | CRM Gateway 复用 CordysCRM 数据权限服务，不绕过原有 Service | 后端实现与单测核对通过 |
| 会话隔离 | 会话和消息按 `userId + organizationId` 查询、更新和软删除 | Conversation Service 4 个单测通过 |
| 写回参数约束 | 后端预留写回工具要求唯一对象、内容和幂等键 | 无效写回请求返回 `WRITEBACK_VALIDATION_FAILED` |
| 当前 Chatflow 只读 | YAML 只调用对象解析与读取链路，不包含 `create-follow-record`、`create-follow-plan` 调用 | 静态搜索和 54 节点校验通过 |
| 不泄露内部协议 | 前端 12 条核心用例检查 `processEvents`、`route_label`、`DIFY_TOOL_TOKEN` 等 | 未发现页面泄露 |
| 失败可恢复 | Dify/网关失败显示失败提示和“重试”按钮 | 多候选场景首次失败后重试成功 |
| 证据不足不编造 | 缺少付款、交付、合同条件时要求补充，不填充虚构事实 | 前端证据不足场景通过 |

后端本轮执行 69 个 `AiDealDesk*` 单测，0 失败、0 错误；本地与 ngrok 固定域名的 Tool API 冒烟均通过。

## 3. 只读与写回边界

当前用户要求“生成跟进记录”“生成跟进计划”时，系统只能：

1. 识别唯一客户或商机。
2. 生成 Markdown 草稿。
3. 标注草稿需人工复核，且没有保存到 CRM。

系统不得声称“已写入”“已保存”“审批通过”“财务已批准”“合同可以直接签署”或“保证上线”。

后端虽然保留 `create-follow-record` 和 `create-follow-plan` 接口，但不代表当前 Chatflow 已交付写回闭环。本轮不使用 `-CreateWriteback`，避免测试本身改变 CRM 数据；测试前后通过数据库状态签名证明无意外写回。

## 4. 数据、权限与会话范围

- 登录页面和历史会话接口要求当前用户身份与组织信息。
- 历史会话、消息查询和删除使用 `userId + organizationId` 双重条件。
- CRM 查询继续通过 CordysCRM 的数据权限服务计算可见范围。
- Dify Cloud 通过固定演示执行身份调用 Tool API，适合当前单租户演示，不等同于每位销售用户的端到端身份透传。
- 外部情报只作为公开信息佐证；必须与 CRM 已确认事实分开表达，不得把公开资料当作客户内部承诺。

## 5. 注入与敏感信息边界

Chatflow 最终回答不得展示 Prompt、模型节点、内部路由、工具令牌、原始协议 JSON 或调试字段。测试用例对以下字符串执行检查：

- `processEvents`
- `route_label`
- `memory_used`
- `quick_answer`
- `full_review`
- `DIFY_TOOL_TOKEN`

当前 YAML 的环境配置包含本地演示 Tool Token 默认值。该值只适合本机和演示环境；如果部署到公开或多人环境，必须改为平台密钥管理、轮换令牌并删除仓库中的默认值。

内部调试信息专项测试没有泄露路由、Agent、节点输出或 Tool Token，但当前回答只泛化为“信息不足”，没有明确解释内部调试信息不对用户开放。该问题属于表达优化，不是数据泄露。

## 6. 失败与降级策略

| 失败场景 | 当前行为 | 演示口径 |
| --- | --- | --- |
| Dify 超时或 504 | 前端显示请求失败，并允许重试 | 记录为稳定性问题；重试成功可继续演示 |
| CRM 读取失败 | 应说明读取失败，不伪装为无数据 | 不允许返回假业务成功 |
| 对象多候选 | 返回真实候选，不猜测具体对象 | 用户选择或输入完整名称后继续 |
| 证据不足 | 输出缺失信息和待确认项 | 合理降级，不视为能力失败 |
| 部分分析缺失 | 输出已完成部分并说明影响 | 不得伪造缺失视角结论 |

## 7. 当前限制与后续要求

以下能力尚未达到生产级，但不作为本轮演示验收失败项：

- Dify Tool API 使用固定演示用户，尚未实现最终用户身份透传和细粒度审计。
- 写回接口虽有参数校验和幂等键要求，前端确认、权限复核、审计日志和失败恢复尚未形成完整闭环。
- 工具令牌仍是单一共享令牌，没有轮换、过期和调用来源限制。
- 没有生产级内容脱敏、数据分级、模型供应商数据保留策略和安全运营监控。
- 模型输出仍有非确定性，复杂评审和外部检索可能出现超时，需要保留重试和演示备用问题。
- 390px 移动端下，全局导航和历史会话占据首屏，当前只建议桌面端演示。

未来启用真实写回前，必须重新验收：对象唯一性、操作权限、人工确认、字段白名单、幂等、审计、失败恢复、撤销策略和跨用户隔离。
