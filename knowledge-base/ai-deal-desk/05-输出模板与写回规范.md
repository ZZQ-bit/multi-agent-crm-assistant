# 多Agent智能助手 - 输出模板与写回规范

本文档用于沉淀 多Agent智能助手 的对话输出结构、前端可渲染卡片、CRM 写回预览、人工确认节点和失败兜底口径。目标是让 Dify Chatflow 输出既适合对话阅读，也能被 CordysCRM 前端稳定渲染和二次操作。

注意：本文档属于知识库规则材料，只提供业务输出结构和风险边界。Agent 行为强约束、内部路由标签隐藏、轻量评审、不默认写回等 Prompt 规则，应维护在 `docs/12-Chatflow Prompt 强约束清单.md` 和 Dify Chatflow 各节点 Prompt 中。

## 1. 输出设计原则

1. 对用户可读：先给业务结论，再给风险依据和下一步。
2. 对前端可渲染：核心内容必须结构化，避免前端解析自然语言。
3. 对 CRM 可写回：写回内容必须是草稿，并经过用户确认。
4. 对权限可控制：不可见字段不得泄露，不可操作按钮应禁用并说明原因。
5. 对失败可恢复：部分失败时仍输出可用结果，并说明影响范围。
6. 对审批有边界：不得输出审批通过、合同可签、财务已通过、交付已承诺等结论。

## 2. 统一响应结构

所有 Agent 输出建议遵循以下结构：

```json
{
  "response_type": "lightweight_deal_review | finance_check | delivery_check | legal_check | sales_summary | followup_plan | writeback_preview | clarification",
  "title": "输出标题",
  "summary": "一句话业务结论",
  "status": "success | partial_success | needs_input | needs_review | failed",
  "severity": "low | medium | high | blocked",
  "agent_trace": [],
  "cards": [],
  "writeback_draft": null,
  "primary_actions": [],
  "missing_info": [],
  "source_refs": []
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| response_type | 输出类型，用于前端选择布局 |
| summary | 面向业务用户的一句话结论 |
| status | 当前任务状态 |
| severity | 整体风险等级 |
| agent_trace | Agent 启用、完成和失败状态，不展示原始思维链 |
| cards | 前端可渲染卡片 |
| writeback_draft | CRM 写回草稿 |
| primary_actions | 可执行操作按钮 |
| missing_info | 缺失字段和追问 |
| source_refs | 引用的 CRM 记录、知识库规则或合同片段 |

## 3. Agent 运行状态展示

前端可展示 Agent 运行过程，但不得展示原始思维链。

推荐展示：

| 状态 | 展示文案 |
| --- | --- |
| route_started | 主 Agent 正在识别任务类型 |
| route_completed | 已识别任务类型和需要启用的专家 Agent |
| agent_started | 某专家 Agent 已启动 |
| agent_completed | 某专家 Agent 已完成 |
| agent_skipped | 某专家 Agent 未启用，原因是本次问题不涉及该领域 |
| tool_started | 正在查询 CRM、知识库或规则工具 |
| tool_completed | 工具调用完成 |
| partial_failed | 部分信息读取失败，结果可信度受影响 |
| synthesis_started | 协调 Agent 正在汇总多方结论 |
| done | 已生成结果 |

`agent_trace` 示例：

```json
[
  {
    "agent_key": "router_agent",
    "agent_name": "主路由 Agent",
    "status": "completed",
    "visible_message": "已识别为复杂商机评审，需要综合销售、财务、交付和合同视角。",
    "started_at": "2026-06-16T10:00:00+08:00",
    "ended_at": "2026-06-16T10:00:03+08:00"
  }
]
```

## 4. 通用卡片结构

所有卡片建议使用以下字段：

```json
{
  "card_type": "risk_summary | agent_result | recommendation | followup_plan | approval_draft | writeback_preview | candidate_list | missing_info",
  "title": "卡片标题",
  "severity": "low | medium | high | blocked",
  "status": "success | needs_input | needs_review",
  "display_fields": [],
  "sections": [],
  "source_refs": [],
  "actions": []
}
```

## 5. 输出模板

### OUT-001 轻量商机评审结论

适用场景：复杂商机评审，通常同时涉及折扣、付款、交付、合同中的两个及以上维度。

必备内容：

1. 结论：用业务语言说明当前方案是否建议继续推进。
2. 关键风险：最多 3 条，覆盖销售、财务、交付、合同中的主要冲突点。
3. 下一步建议：一线销售可执行的 2 到 4 个动作。
4. 需要复核的角色：销售主管、财务、交付、商务/法务等。
5. 缺失信息：如果关键字段不足，说明缺失内容对结论的影响。

不默认包含 CRM 写回草稿。只有用户明确要求保存、写入 CRM、生成跟进记录或创建跟进计划时，才进入写回预览与人工确认。

### OUT-002 财务风控判断

适用场景：`route_label=finance_task`

必备字段：

- `conclusion`
- `risk_level`
- `matched_rules`
- `finance_risks`
- `recommended_terms`
- `required_approvers`
- `missing_finance_info`

不得输出：

- 折扣已批准
- 财务已通过
- 可以直接承诺账期

### OUT-003 交付可行性判断

适用场景：`route_label=delivery_task`

必备字段：

- `conclusion`
- `risk_level`
- `scope_assumption`
- `delivery_risks`
- `customer_prerequisites`
- `recommended_delivery_plan`
- `required_reviewers`

不得输出：

- 保证上线日期
- 交付资源已锁定
- 定制需求已确认包含在标准交付中

### OUT-004 合同风险判断

适用场景：`route_label=legal_task`

必备字段：

- `conclusion`
- `risk_level`
- `clause_risks`
- `suggested_revision_points`
- `required_legal_review`
- `missing_contract_info`

不得输出：

- 合同可以签
- 法务已通过
- 条款可直接接受

### OUT-005 商机总结

适用场景：用户要求总结客户、商机或近期进展。

必备内容：

1. 商机概览：客户、金额、阶段、负责人、产品范围。
2. 近期进展：最近沟通、客户反馈、关键人状态。
3. 商机健康度：预算、决策链、竞品、采购流程、跟进节奏。
4. 主要风险：列出可被销售处理的风险。
5. 下一步建议：动作、对象、时间、目标。
6. 可写回草稿：客观记录，不写审批结论。

### OUT-006 客户/商机查询候选卡

适用场景：查询命中多个对象或上下文不唯一。

### OUT-008 联系人角色地图卡

## 6. CRM 写回白名单

允许写回：

| 写回类型 | 允许字段 |
| --- | --- |
| 跟进记录 | 客户诉求、沟通摘要、AI 初步风险、待确认事项、下一步动作 |
| 跟进计划 | owner、对象、时间、方式、目标、成功标准 |
| AI 评审摘要 | 评审结论、风险等级、命中规则、需复核角色 |
| 审批说明草稿 | 申请事项、业务理由、风险缓释、替代方案、所需材料 |

禁止自动写回：

- 审批通过结论
- 正式报价承诺
- 合同可签结论
- 交付日期承诺
- 财务授信结论
- 客户未经确认的口头承诺
- 覆盖已有关键字段但未展示差异的内容

## 7. 写回前校验

写回前必须校验：

| 校验项 | 规则 |
| --- | --- |
| 对象唯一性 | 必须绑定唯一 customer_id 或 opportunity_id |
| 字段白名单 | 只能写入允许字段 |
| 用户确认 | 必须点击确认，不能自动保存 |
| 差异预览 | 覆盖已有内容前展示差异 |
| 禁用词检查 | 不得包含“已批准”“已承诺”“合同可签”等越权结论 |
| 幂等处理 | 重试时不得重复创建记录 |
| 权限检查 | 当前用户必须有目标对象写权限 |

## 8. 操作按钮规范

| action_key | 展示文案 | 启用条件 |
| --- | --- | --- |
| select_candidate | 选择该客户/商机 | 查询结果多于 1 个 |
| generate_full_review | 发起商机评审 | 已绑定商机，且涉及两类以上约束 |
| generate_followup_plan | 生成下一步计划 | 已绑定客户或商机 |
| create_writeback_draft | 生成写回草稿 | 有可写回内容 |
| save_to_crm | 保存到 CRM | 写回草稿通过校验且用户确认 |
| copy_approval_draft | 复制审批说明 | 审批草稿已生成 |
| ask_missing_info | 补充信息 | 存在阻塞字段 |
| retry_failed_tool | 重试查询 | 工具调用失败且可重试 |

按钮字段示例：

```json
{
  "action_key": "save_to_crm",
  "label": "保存到 CRM",
  "enabled": false,
  "disabled_reason": "缺少唯一商机编号，无法写回",
  "requires_confirm": true,
  "next_state": "writeback_confirm"
}
```

## 9. 失败兜底规范

| 场景 | 输出策略 |
| --- | --- |
| CRM 查询失败 | 说明查询失败，保留用户输入分析，提示重试或手动选择对象 |
| 知识库召回为空 | 输出基于通用规则的初步判断，并标记需人工复核 |
| 某个专家 Agent 失败 | 输出其他 Agent 结果，标记 partial_success 和失败影响 |
| 对象不唯一 | 输出候选列表，不做写回 |
| 缺少关键字段 | 输出缺失字段和追问，不输出低风险结论 |
| 写回失败 | 保留草稿，提示重试，不重复创建记录 |

## 10. 禁用边界

所有输出不得：

- 展示模型原始思维链。
- 把 Agent 建议包装成审批结论。
- 在缺少关键字段时输出低风险。
- 用自然语言隐藏越权承诺。
- 自动写回未经用户确认的内容。
- 泄露用户无权限查看的客户、合同或财务字段。
