# 商机评审改造点设计(2026-07-01)

> 状态:方案阶段,等待用户评审
> 作者:Claude(基于 brainstorming skill 多轮澄清)
> 项目:CordysCRM(2026 Q3 改造项)

## 1. 背景与目标

### 1.1 业务背景

CordysCRM 当前的销售能力以 多Agent智能助手 对话式评审为主(v3 已在使用)。销售在商机详情页工作时,需要"一步跳出"看到一份**结构化的多维度商机评审报告**,而不是再开一个新会话问 多Agent智能助手。

具体来说:

- 商机详情页缺少评审入口
- 多Agent智能助手 是对话式、上下文累积的,适合追问 / 写回,但不适合"快速给一份报告作为档案"
- 用户需要查看的报告包含两部分:**商机自身分析**(CRM 内部数据)+ **所属客户分析**(CRM 数据 + 联网搜索公开信息)

### 1.2 目标

在商机维度新增一个**异步、结构化、只读**的评审报告能力,让销售可以在不开启对话、不写回 CRM 的前提下,获得一份可以反复查看的多维度分析。

### 1.3 非目标

- **不**替代 多Agent智能助手 的对话式评审
- **不**支持追问 / 多轮对话
- **不**写回 CRM(不存跟进、不存附件)
- **不**保留历史报告版本(只保最新)
- **不**支持分享 / 导出
- **不**覆盖 v3 已有的对话式评审场景(用户已可在 多Agent智能助手 完成的事)

**v4.2 已知限制**(由数据可达性决定):
- **联网搜索可能失败**(通用网络问题),失败时新闻舆情模块不渲染,不阻塞报告
- **不做**行业聚合对比("近 6 个月同行业均值"),无数据仓库支撑
- **不渲染**合同条款明细(合同字段大量是自定义动态字段,渲染成本高,v4.3 再设计)
- **不展示**交付团队排期(NPS、人数等,CRM 没有这些字段)
- **不展示**客户财务数据(营收/净利率,QCC 只到基础工商信息)

## 2. 设计决策汇总

| 维度 | 决策 |
|------|------|
| 能力定位 | 独立 chatflow(v3 思路复用,但新建一个),后端工具能复用就复用 |
| 报告形态 | 只读展示,不写回 CRM |
| 报告内容 | **3 大高价值模块**:AI 分析判断 + 客户新闻舆情(联网) + 行动建议;商机基础信息不重复(头部一行) |

#### 4.1.3 字段可达性约束

- `context.*` 5 个字段对应 `OpportunityDetail` 固定字段(只用于头部一行展示,不重复报告正文)
- `ai_analysis.key_points[]` 由 LLM 综合 CRM 商机字段 + 跟进数据归纳,3-5 条
- `ai_analysis.evidence_refs[]` 每条都对应可追溯的 CRM 数据点(阶段停留天数/最近跟进时间等)
- `customer_profile.qcc.*` 通过 `cn.cordys.crm.integration.qcc.InfoData` 直接对应
- `external_news[]` 由 Dify 工具节点调通用联网搜索(返回 3-5 条相关新闻);**如联网搜索失败,字段为 `[]` 且不渲染该模块**
- `recommendations[]` 由 LLM 基于 `ai_analysis` + `external_news` 综合生成 2-4 条可执行建议

#### 4.1.4 节点划分

1. **参数解析节点**:校验入参
2. **商机上下文节点**:调 `get-opportunity-context` + `getOptFollowRecordList`,产出 `context` + 跟进数据(供后续 LLM 用)
3. **客户工商节点**:调 `cn.cordys.crm.integration.qcc` + `get-customer-context`,产出 `customer_profile.qcc`
4. **联网搜索节点**:调 Dify 工具节点的联网搜索(如 `tavily` / `web_search`),以客户名为关键词,产出 `external_news[]`
5. **AI 综合判断节点**:LLM 综合上述数据,产出 `ai_analysis`(summary + key_points + evidence_refs)
6. **建议生成节点**:LLM 基于 `ai_analysis` + `external_news`,产出 `recommendations[]`
7. **合成报告节点**:合并上述结构输出

#### 4.1.5 v3 复用点

- CRM 工具集整体复用:`get-opportunity-context` / `get-customer-context` / `getOptFollowRecordList` 已在 v3 `AiDealDeskToolController` 中
- QCC 集成(`cn.cordys.crm.integration.qcc.InfoData`)直接复用
- LLM 模型 / Prompt 风格保持 v3 范式
- **v4.2 重新引入联网搜索**:用 Dify 工具节点(`tavily` / `web_search`),由 Dify 侧配置 API Key 与重试,**不**经 CordysCRM 后端

| 报告字段 | 数据源 | 可达性 | 降级策略 |
|---------|--------|--------|---------|

### 4.2 CordysCRM 后端

#### 4.2.1 数据模型

新增一张轻量表,只保最新一份:

采用三张表:**主表 + 报告内容表 + Dify 调用日志表**。

主表 `opportunity_review` 只存状态与元数据,把报告内容和调用日志拆出去,既避免主表行过大,也便于将来扩展到客户/合同评审能力。

```sql
-- 主表:状态 + 元数据(预留多实体评审)
CREATE TABLE opportunity_review (
  id                    BIGINT       PRIMARY KEY AUTO_INCREMENT,
  entity_type           ENUM('opportunity','customer','contract') NOT NULL,
  entity_id             VARCHAR(64)  NOT NULL,                    -- 例如 opportunity_id
  status                ENUM('pending','success','failed') NOT NULL,
  failure_reason        TEXT         NULL,                        -- 面向用户,简明
  generated_at          DATETIME     NULL,                        -- 报告生成完成时间
  dify_task_id          VARCHAR(128) NULL,
  chatflow_version      VARCHAR(32)  NULL,
  regen_count           INT          DEFAULT 0,                   -- 用户主动重新评审次数(仅计数,无历史)
  created_by            VARCHAR(64)  NOT NULL,
  created_at            DATETIME     NOT NULL,
  updated_at            DATETIME     NOT NULL,
  UNIQUE KEY uk_entity (entity_type, entity_id),                  -- 每实体只一条
  INDEX idx_status_updated (status, updated_at)
);

-- 报告内容表:与主表 1:1,只存报告 JSON
CREATE TABLE opportunity_review_report (
  review_id             BIGINT       PRIMARY KEY,
  report_json           JSON         NOT NULL,
  INDEX idx_generated (review_id)
);

-- Dify 调用日志表:运营/开发排查用,前端不展示
CREATE TABLE opportunity_review_dify_log (
  id                    BIGINT       PRIMARY KEY AUTO_INCREMENT,
  review_id             BIGINT       NOT NULL,
  event                 ENUM('workflow_started','workflow_succeeded','workflow_failed','workflow_timeout') NOT NULL,
  duration_ms           INT          NULL,                        -- workflow_started 时为 NULL,succeeded/failed/timeout 时为总耗时
  error_snippet         TEXT         NULL,                        -- failed/timeout 时记录关键错误信息
  occurred_at           DATETIME     NOT NULL,
  INDEX idx_review (review_id, occurred_at)
);
```

**只存最新一份 + 多实体可扩展**:
- `UNIQUE KEY (entity_type, entity_id)` 保证每实体只有一条最新记录
- 重新评审即 `UPDATE` 主表 + `UPDATE` 报告内容表,**不保留历史**
- `entity_type` 预留使客户/合同评审能力未来可零成本接入(共用此结构)
- Dify 节点级日志单独落表,前端不展示技术细节

#### 4.2.2 API 契约

| 方法 | 路径 | 用途 | 入参 | 出参 |
|------|------|------|------|------|
| POST | `/api/opportunities/:id/review` | 启动评审 | `{request_id?}` | `{status: 'pending', dify_task_id}` |
| GET | `/api/opportunities/:id/review` | 查最新 | — | `{status, report?, generated_at?}` |
| POST | `/api/opportunities/:id/review/regenerate` | 重新评审 | `{request_id?}` | 同 POST |
| GET | `/api/opportunities/:id/review/stream` | SSE 流式状态推送(可选) | — | SSE 事件 |

**幂等性**:

- POST 启动时,若当前记录 `status='pending'`,**直接返回现有 task_id**(即"未完成不会有重新评审入口"的实现兜底:即便前端绕过,后端也不会启动第二个任务)
- POST regenerate 同理,只在 `status='success' | 'failed'` 时才真正重置 `status='pending'` 并发起 Dify

**与 多Agent智能助手 接口的关系**:

- 路径前缀完全独立:`/api/opportunities/:id/review/*` ≠ 多Agent智能助手 任意路径
- 共享底层的 CRM MCP 工具、数据源、消息中心(均为后端内部组件,不暴露)
- 外部接口不互相影响,符合 v1/v2/v3 切换的"外部接口必须完全一致"的延伸精神(虽然本次不是版本切换,但保持同样的隔离纪律)

#### 4.2.3 状态机

```
            ┌──────────┐
            │ (无记录) │
            └────┬─────┘
                 │ POST /review
                 ▼
          ┌──────────┐  Dify 成功   ┌──────────┐
          │ pending  │ ───────────► │ success  │
          └────┬─────┘              └────┬─────┘
               │ Dify 失败               │ POST /regenerate
               ▼                         ▼
          ┌──────────┐              ┌──────────┐
          │  failed  │ ────────────►│ pending  │
          └──────────┘ POST         └──────────┘
                     /regenerate
```

#### 4.2.4 Dify 集成(异步触发方式)

**采用方案**:Dify workflow 异步运行 API

- 后端调 `POST /v1/workflows/run` (async=true),拿到 `workflow_run_id`
- 后端启动一个 worker:`while (status not terminal) { sleep 2s; GET /v1/workflows/run/:id }`
- 终端状态(`succeeded` / `failed` / `stopped`)时,落库 + 推送消息
- worker 失败/超时(默认 5 分钟无更新)→ 状态置 `failed`,failure_reason 记录

**重试策略(后端不参与节点级重试,只对整次 chatflow 重试)**:

联网重试由 Dify 工具节点自身配置(API Key、超时、重试次数都在 Dify 侧管理),CordysCRM 后端**不感知联网细节**,只对整次 chatflow 结果负责。

| 维度 | 责任方 | 说明 |
|------|--------|------|
| 联网节点级重试 | Dify 工具节点 | 在 Dify 工作流编辑器里配置(默认 3 次 + 指数退避),无需后端参与 |
| 整次 chatflow 重试 | 用户手动 | 用户在报告页点"重新评审"即可;后端只是把状态重置为 pending 并调一次 Dify |
| 整次超时兜底 | 后端 worker | 默认 5 分钟无更新 → status=failed,failure_reason="chatflow 超时,可手动重试" |

后端 worker **不再记录 attempt 序列**——Dify 工具节点自身的重试对后端不可见。日志表 `opportunity_review_dify_log` 改为只记录整次 chatflow 的状态变化(每次 status 切换写一条),不再按节点 attempt 记。

**优点**:与 v3 一致(假设 v3 也是这套);不依赖 webhook,部署简单。

**可选优化(本次不做)**:Dify 完成后通过 webhook 回调 → 后端,再由后端推前端。比轮询省 50% 延迟,但部署要 Dify 端配置外网地址。

**状态切换**:

**重复评审处理**:

- 用户在报告页"重新评审" → 立即把 status 置为 pending,UI 切到骨架屏
- 旧报告**不可查看**(UI 层强制,符合"未完成不会有重新评审入口"的精神)
- 报告页停留时,后端 SSE 或前端轮询 2s,状态变化自动切

**轮询策略**:

- 进入页面时立即 GET 一次
- status=pending:每 2s 轮询一次,最多 5 分钟(后端超时会置 failed)
- status=success/failed:停止轮询,等用户主动操作

**消息通知接入**:

- 报告生成成功 → 复用 CordysCRM 现有的消息中心(铃铛),新增 `deal_desk_review_done` 通知类型
- 通知点击 → 跳转 `/opportunity/:id/review` 二级页
- 不在通知里贴报告全文

## 5. 边界与异常

### 5.1 边界场景

| 场景 | 行为 |
|------|------|
| 用户重复点"发起评审",已有 pending | 后端幂等返回,前端跳转到的页面已 polling |
| 用户在二级页"重新评审",旧报告还在 | 立即重置 status=pending,UI 切骨架屏,旧报告不再可见 |
| 生成超时(> 5 分钟) | 后端 worker 主动置 failed,前端看到"生成失败,请重试";`opportunity_review_dify_log` 保留最后节点状态 |
| 联网搜索部分失败 | chatflow 报告里客户分析的 `external_research` 仅含成功的部分;LLM 在 prompt 中已提示"部分外部信息不可用",报告相应维度的 `level` 标记为 `info`,UI 自动渲染"部分外部资料缺失" |
| 联网搜索全部失败(Dify 工具节点重试用尽) | chatflow 仍可能产出"客户分析"骨架(只有 CRM 内部信息);若 Dify 把这种情况标为整次失败 → 主表 `status=failed`,`failure_reason="客户公开信息搜集失败,可手动重试"`;运维查 `opportunity_review_dify_log` 看到 `workflow_failed` 一条记录 |
| 商机被删除 | 报告表记录保留(避免 FK 复杂),但 UI 不再展示入口 |
| 用户无 CRM 权限 | 后端鉴权失败 → 返回 403,前端现有权限框架接管 |
| Dify 端宕机 | 后端调 API 立即失败 → status=failed,reason="chatflow unavailable" |

### 5.2 错误码

```
REVIEW_PENDING        当前已有评审在进行,直接返回 task_id
REVIEW_NOT_FOUND      未发起过评审,需先 POST
REVIEW_DIFY_TIMEOUT   chatflow 超时
REVIEW_DIFY_UNAVAILABLE   Dify 服务不可用
REVIEW_INTERNAL       其他内部错误
```

### 5.3 安全

- 所有 API 走 CordysCRM 现有鉴权(cookie + RBAC)
- 报告 JSON 中的 `external_research.url` 不直接渲染为 `<a href>`,而是 `<a target=_blank rel=noopener>`,防止钓鱼
- 报告本身不做 XSS 过滤(因为内容由 LLM + CRM 内部数据产出),但前端用结构化 JSON 渲染(不用 v-html),从源头规避
