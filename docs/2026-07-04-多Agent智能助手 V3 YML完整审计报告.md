# 多Agent智能助手 V3 YML 完整审计报告

审计对象：`chatflows/ai-deal-desk-v3.example.yml`

最近复核时间：2026-07-06

## 1. 总体结论

当前 YML 可以作为多Agent智能助手 V3 Stable Enhanced 的主 Chatflow 基线继续使用。文件能被 YAML 解析器正常读取，当前图结构包含 54 个节点、75 条边。

这版已经从早期“先固定路由、各分支独立回答”的结构，调整为：

```text
附件与输入解析
  -> Planner 任务理解与规划
  -> 任务计划标准化
  -> 任务类型路由
  -> 证据收集调度
  -> CRM / 知识库 / 外部情报 / 附件证据按需收集
  -> 证据汇总与缺口判断
  -> 业务 Agent 路由
  -> 专项结论汇合
  -> CRM 业务回答整合
  -> 输出协议适配
```

当前版本重点是只读演示稳定性：AI 负责查询、判断、建议和草稿生成，不直接替用户审批，也不调用 CRM 写回工具。

## 3. 当前能力覆盖

| 能力 | 当前链路 |
| --- | --- |
| 普通问答 | `task_type_gate` 分流到轻量回答，再进入统一协议适配 |
| 图片问答 | 附件图片识别后进入图片直接回答，再进入统一协议适配 |
| 客户/商机查询 | 证据调度按需调用 CRM 读取，并由 CRM 轻量读取回答或业务回答整合输出 |
| 商机总结/销售建议 | CRM 证据 + 销售协同 Agent + 统一业务回答 |
| 财务/交付/合同专项判断 | CRM/知识库证据 + 对应专项 Agent + 统一业务回答 |
| 复杂商机评审 | 销售、财务、交付、合同多 Agent 按需并行，再统一汇总 |
| 经营分析 | CRM 统计证据 + 经营分析 Agent |
| 外部情报 | 仅在 Planner 判断需要时触发外部情报 Agent |
| 写回类诉求 | 生成可复制草稿，不调用真实写回接口 |

## 4. 风险与注意事项

1. **知识库 dataset 需要导入后确认。** Dify 跨工作区导入时，知识库 ID 通常不能直接复用，需要在目标空间重新绑定规则知识库。
2. **模型与工具 Key 不能进入公开仓库。** 模型供应商 Key、Tavily Key、Dify App Key、Tool Token 都必须通过环境变量或 Dify 控制台配置。
3. **固定 ngrok 域名不应写入公开版。** 公开文档只保留 `<your-ngrok-domain>` 这类占位符。
4. **当前不做真实写回。** 前端协议可保留写回兼容字段，但演示口径应明确“生成草稿、人工复制或后续阶段写回”。
5. **旧测试文档未同步修改。** 人工测试集按用户要求保持不动；如果后续要正式验收，应另开任务更新测试集。

## 5. 上线前建议清单

1. 在 Dify Cloud 重新导入并发布当前 `chatflows/ai-deal-desk-v3.example.yml`。
2. 检查 Dify 环境变量：
   - `CRM_TOOL_BASE_URL`
   - `DIFY_TOOL_TOKEN`

3. 检查规则知识库是否在目标 Dify 工作区重新绑定。
4. 做 6 类回归：
   - 普通问答
   - 图片问答
   - 客户/商机查询与多结果澄清
   - 商机总结或销售建议
   - 复杂商机评审
   - 写回类诉求降级为可复制草稿

## 6. 最终判断

当前 YML 的结构已经与作品集主线一致：以 CRM 业务对象为基础，以 Planner 决策和证据台账为中枢，以多 Agent 分工体现复杂判断能力，以统一协议适配前端展示。

后续优先完成部署演示和公开版说明，再考虑是否恢复真实 CRM 写回。
