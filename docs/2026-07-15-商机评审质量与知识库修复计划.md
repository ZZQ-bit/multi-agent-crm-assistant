# 商机评审质量与知识库修复计划

更新时间：2026-07-15

## 总结

当前重复来自两处叠加：

- Planner 的成功标准要求“四个角度分别输出结论、风险、建议，再给综合结论”。
- 最终综合节点虽要求去重，但又允许完整评审按四个领域展开，模型因此生成“四段分述 + 再综合一次”。

知识库还有独立问题：

- 最近一次完整评审被 Planner 判定为 `need_knowledge=false`，完全没有检索知识库。
- 历史日志中，一次包含“20% 首付、验收后 90 天、数据安全承诺”的高度相关查询仍返回空数组，说明 Dify 数据集绑定、索引、模型额度或检索配置曾经存在问题。
- 知识库原文内容本身足以支撑当前作品集场景，已经包含财务、交付、合同规则、轻量评审模板和人工确认边界。
- 当前 Chatflow 的检索输入在 CRM 证据返回前生成，只拼接用户原话、固定任务词和 Planner 回答目标；历史输入存在重复、主题堆叠和无关描述混入的问题。

本轮已单独将 Reranker 更新为 `Qwen/Qwen3-Reranker-4B` 并验证直接检索效果明显改善。后续阶段不同时更换 Planner、领域 Agent 或综合回答模型，先独立修复召回输入、知识库内容和回答质量，保证问题可归因。

所有实施继续遵循：

- 一次只实施一个阶段。
- 每阶段完成后暂停，由用户导入、发布并验收。
- 用户明确同意后才能进入下一阶段。
- 耗时不设置强制门槛，但继续记录总耗时以及所有超过 2 秒的节点。

## 分阶段实施

### 阶段 1：修复并固定 Dify 知识库健康状态

状态：已完成。用户已确认知识库索引、检索配置和 Reranker 调整通过验收，同意进入阶段 2。

使用 [AI Deal Desk 规则知识库 - Dify 单页版.md](../knowledge-base/ai-deal-desk/AI%20Deal%20Desk%20规则知识库%20-%20Dify%20单页版.md) 确认当前 Dify 数据集：

- 文档已完成索引。
- 分段数量大于 0。
- Chatflow 的知识检索节点绑定当前数据集 ID。
- Dify 模型额度正常，Embedding 和 Rerank 调用没有失败。

当前配置基线：

- 分段模式：General。
- 分段符：双换行。
- 最大分段长度：1024。
- 重叠长度：100。
- 索引模式：高质量。
- Embedding：`BAAI/bge-m3`。
- 检索方式：混合检索。
- Rerank：`Qwen/Qwen3-Reranker-4B`。
- Top K：8。
- Score 阈值：关闭。

在 Dify 知识库测试界面验证：

1. “88 万、75 折、30% 首付”能召回 `FIN-004/005/008/009` 中的主要相关规则。
2. “2 周上线标准版”能召回 `DEL-002`。
3. “延期赔付、验收样本、赔付上限”能召回 `LEG-001/009/010` 或 `DEL-031`。
4. “完整商机评审输出”能召回 `OUT-001`、`CORDYS-ROLE-010/012`。

已经确认：Qwen3-Reranker-4B 能将折扣换条件、深度折扣和大额深折扣稳定排到前列，优于原 `bge-reranker-v2-m3`。如果上述直接检索仍出现高度相关问题零召回，应先检查额度、索引和数据集绑定；确认不可恢复后重建数据集并更新 Chatflow 的 `dataset_ids`，不继续修改 Prompt。

阶段 1 完成后暂停，由用户验收并明确同意后才能进入阶段 2。

### 阶段 2：让 Planner 语义决定知识库，并基于 CRM/附件证据生成检索查询

状态：已完成并由用户验收。Planner 的知识策略、CRM/附件证据驱动查询、并行关系、跳过路径和查询降噪均已在线验证。已知限制是当前单次综合查询在完整评审中可能更偏向最显著的财务主题；用户接受该限制并同意进入阶段 3，后续不在本阶段增加多查询循环。

#### 2.1 Planner 保留语义决策权

不按 `task_type`、关键词或代码规则强制检索。Planner 根据语义判断：

- 回答需要内部政策、风险规则、业务基线、审批或责任边界时，选择知识库。
- 纯 CRM 事实读取、普通摘要、普通问答或不需要规则依据时，不查询知识库。

Planner 内部 JSON 将原布尔判断扩展为：

```json
{
  "knowledge_policy": "none | required | conditional",
  "knowledge_goal": "本轮希望知识库提供什么规则依据"
}
```

- `none`：直接跳过知识库。
- `required`：任务明确需要内部规则、审批要求、业务基线或责任边界，缺少规则依据就无法可靠完成判断。
- `conditional`：是否检索取决于 CRM 或附件中是否出现会实质影响回答的业务事实。

完整商机评审通常优先选择 `conditional`；当用户明确要求按内部规则评审，或 Planner 判断缺少规则依据就不能负责任地给出结论时，再选择 `required`。

Planner 负责决定策略和目标，但不直接生成最终 `knowledge_query`，因为此时尚未读取 CRM 和附件事实。

#### 2.2 CRM、附件和外部情报保持并行

```text
Planner - 任务理解与规划
  │
  ├───────────────┬────────────────┐
  ↓               ↓                ↓
CRM 数据读取     附件证据整理      外部情报查询
或快速跳过       或快速跳过         或快速跳过
  │               │                │
  └───────┬───────┘                │
          ↓                        │
     检索上下文准备                 │
          ↓                        │
  知识检索决策与查询生成             │
          ↓                        │
      知识检索结果校验               │
          ↓                        │
      是否检索知识库                 │
        ├─ 跳过                     │
        └─ 知识库检索               │
          │                        │
          └──────────┬─────────────┘
                     ↓
                  证据汇总
                     ↓
                  领域 Agent
```

要求：

- CRM 数据读取、附件证据和外部情报同时启动，互不等待。
- 外部情报不参与内部规则检索词生成，不成为知识库的前置依赖。
- 有 CRM 时，检索词使用与本轮任务有关的 CRM 事实摘要。
- 有附件时，检索词使用与本轮任务有关的附件条款或业务事实摘要。
- CRM 和附件都不需要时，纯规则问题直接使用用户问题和 Planner 回答目标生成查询。
- `knowledge_policy=none` 时直接跳过新增的检索决策 LLM。

#### 2.3 生成最终 knowledge_query

新增 LLM 节点 `知识检索决策与查询生成`，输入：

- 用户原始问题。
- Planner 的任务类型、回答目标、`knowledge_policy`、`knowledge_goal`。
- CRM 证据摘要或跳过/失败状态。
- 附件证据摘要或跳过/失败状态。

输出：

```json
{
  "need_knowledge": true,
  "knowledge_query": "88万元商机，75折，首付30%，尾款依赖验收，检索折扣、付款和回款风险规则",
  "focus_domains": ["finance", "legal"],
  "facts_used": ["金额88万元", "折扣75折", "首付30%", "尾款依赖验收"],
  "reason": "需要内部财务和合同规则支持本轮判断"
}
```

查询规则：

- 删除现有查询改写节点中的固定 `TASK_TERMS` 词表。
- 查询长度限制在 260 字符以内，但不根据关键词裁剪用户原意。
- 只保留与本轮判断直接相关的已确认事实、约束关系和规则主题。
- 客户名、商机名、负责人、联系人、普通销售阶段和泛化标签默认不进入规则查询；只有规则适用性确实依赖时才保留。
- 完整评审优先保留金额、折扣、付款、账期、交付周期与范围、验收样本、试运行周期、退款/赔付条件和责任边界等可影响规则适用性的原始事实，避免笼统写成“验收边界”或“合同风险”。
- 尚未确认的客户诉求必须保留“可能、若坚持、备选方案”等不确定性，不得改写成已确认条款。
- 不传入完整 CRM JSON、全部跟进历史、重复问题、页面调试信息或无关任务标签。
- 不编造 CRM 或附件中不存在的事实。
- 第一版只生成一个高质量综合查询，不增加多查询循环。
- `reason` 只保留一句审计说明，不输出思维链。

现有 `知识检索词整理` 改为 `知识检索结果校验`，只负责结构校验、枚举校验、空值处理、去重、260 字截断和失败兜底，不再承担业务语义扩写。

缺失或失败时：

- `none`：直接跳过。
- `required`：使用“用户问题 + knowledge_goal + 已取得的简要事实”生成降级查询。
- `conditional`：在 `knowledge_goal` 非空时使用同样的降级查询；否则跳过并记录决策失败。
- CRM 失败但用户已经给出可判断事实时，使用用户事实继续查询，同时保留 CRM 失败状态。

#### 2.4 知识证据包

知识证据包最多保留 8 条去重结果，每条保留：

- 规则编号和规则名称。
- 正文片段。
- 相关度。
- 文档来源。

零召回时明确记录“未召回规则”；领域 Agent 和综合节点不得伪造规则编号、来源或政策内容。

#### 2.5 阶段验收

1. CRM 事实查询：Planner 选择 `none`，不调用新增检索决策 LLM和知识库。
2. 纯规则问题：不读 CRM，直接生成语义查询并检索知识库。
3. 财务单项判断：CRM 返回后，查询包含实际金额、折扣、首付、付款或验收事实中的相关子集。
4. 完整评审：CRM 返回后精准查询；未要求公开资料时不运行外部情报。
5. 附件合同评审：CRM 与附件并行，查询使用附件实际条款和可用 CRM 事实。
6. 同时要求公开信息和内部规则：外部情报与 CRM/附件链并行，知识检索不等待外部情报。
7. CRM 失败但用户给出 75 折和 30% 首付：继续检索并标记 CRM 证据失败。
8. 普通话术或闲聊：不运行新增检索节点。

验收标准：

- Planner 的知识库选择来自语义判断，代码中不存在关键词路由或硬评分。
- 检索输入不再出现历史日志中的重复问题、任务标签堆叠和无关演示描述。
- 查询不超过 260 字，不包含完整 CRM JSON，不虚构业务数值。
- CRM、附件和外部情报能够并行启动，外部情报不成为知识检索的前置依赖。
- 需要 CRM/知识库、仅 CRM、仅知识库、两者都不需要四类路径均能进入证据汇总。
- 本阶段不修改领域 Agent、最终综合 Prompt 或主模型。

阶段 2 完成后暂停，由用户验收并明确同意后才能进入阶段 3。

### 阶段 3：治理知识库内容，压缩领域 Agent，重构综合回答

状态：已完成最终在线验收，等待用户确认是否进入阶段 4。领域 Agent 压缩、主题驱动组织、四视角矩阵、唯一下一步清单、风险一致性、参考规则边界和单次责任边界均已生效；普通评审与明确四视角评审均不再生成第二个统一结论。

#### 3.1 知识库内容治理

- 将知识库内容分为硬责任边界、正式业务政策和参考风险信号。
- 保留对象唯一性、人工确认、不得越权承诺、不得伪造证据和写回安全等硬边界。
- 正式政策应具备来源、版本、适用范围和责任人；无法确认来源的折扣、SLA、渠道、续费等硬数字标记为演示假设或参考信号。
- 规则自身的风险等级只作为领域 Agent 的证据，不自动等同于整体商机风险。
- 将统一响应 JSON、前端卡片字段、节点状态和 Agent 输出格式移出业务规则召回层，由 Chatflow 和协议文档管理。
- 更新完整评审规范：默认保留最重要的 3 条问题；确有独立重大问题时最多 5 条。
- 当前继续使用 Dify 单页知识库，优化段落标题、规则主题和近义规则重复；本阶段不同时改为多数据集架构。

#### 3.2 领域 Agent 内部输出合同

销售、财务、交付、合同四个领域 Agent 继续独立判断，但内部输出改为紧凑结构：

- 领域判断。
- 最多 2 条该领域独有发现。
- CRM、知识规则或推断依据。
- 最多 2 个动作。
- 缺失信息和需人工确认事项。

领域 Agent 不再重复商机金额、阶段、客户、负责人、联系人等公共概览，只引用对本领域结论必要的证据。领域输出是综合节点的内部材料，不各自生成一篇完整用户报告。

#### 3.3 综合回答

综合节点采用“主题驱动自适应”：

- 先用 1–2 句话给出整体判断。
- 围绕当前最重要的 3–5 个业务问题组织内容。
- 合并跨领域重复事实、风险和动作。
- 给出 3–5 个可执行下一步。
- 缺失信息只展示会实质改变结论的部分。

用户明确要求四个视角时，优先使用一个紧凑矩阵展示各视角的独有判断，不再生成四个完整章节。表格仅用于三项以上可比较信息；图表仅用于真实的趋势、分布、漏斗或阶段数据，定性评审不强行生成图表。

风险等级由 LLM 综合判断影响、发生可能性和可缓释性；优先级综合判断业务价值、时间窗口和推进成熟度。不使用分数、关键词命中或代码评分。同一轮风险和优先级标签必须前后一致。

本阶段：

- 不做数据脱敏。
- 最终只保留一次责任边界：AI 输出是内部评审建议，折扣、付款、交付承诺和合同条款需相应负责人确认。
- 不增加第二个质量检查 LLM 节点，避免再次增加串行耗时。
- 回答长度不设硬性字符限制，以信息价值、可读性和无赘述为准。

#### 3.4 最终在线验收记录

最终发布版本于 2026-07-15 使用两条完整评审用例验收：

1. 自然表达完整评审：运行成功，40 个节点，工作流耗时 98.63 秒，最终回答 2522 字符。回答包含开头唯一结论、4 个具有独立业务价值的关键议题、唯一一组 5 个下一步、关键待确认信息和一次责任边界；议题分析结束后直接进入下一步，没有第二个综合结论。
2. 明确要求销售、财务、交付、合同四视角：运行成功，40 个节点，工作流耗时 89.16 秒，最终回答 1633 字符。回答使用“视角 / 独有判断 / 必要证据”三列矩阵，矩阵后直接进入唯一下一步清单，没有“统一推进结论”或“统一交易结构”二次复述；数值规则明确标记为内部评审参考，不表述为已批准政策。

最终两条用例中超过 2 秒的节点如下：

| 节点 | 自然完整评审 | 四视角评审 |
|---|---:|---:|
| CRM 业务回答生成 | 35.07s | 27.47s |
| Planner - 任务理解与规划 | 21.95s | 20.08s |
| 销售协同 Agent | 23.95s | 9.95s |
| 财务判断 Agent | 11.36s | 12.10s |
| 交付判断 Agent | 10.85s | 13.35s |
| 合同判断 Agent | 11.79s | 8.89s |
| 知识检索决策与查询生成 | 7.69s | 10.45s |
| 知识库检索 | 5.31s | 4.27s |

其余节点均低于 2 秒。阶段 3 的回答质量目标已经通过；模型速度优化继续作为后续独立变量，不与本阶段 Prompt 和知识规则改造混合归因。

阶段 3 完成后暂停，由用户验收并明确同意后才能进入阶段 4。

### 阶段 4：知识库与回答质量回归

状态：五类核心回归已完成，等待用户验收。CRM 事实查询、财务、交付、合同和完整评审的节点路由均符合预期；AI 新生成的付款比例已被约束为“备选谈判草案，待财务复核”，不得表述成已批准底线。对象名称与后续描述粘连属于低优先级边界问题，本阶段不修改 CRM 解析逻辑，也不增加 Planner 语义重试；Planner 已恢复为最多 2 次迭代。CRM 快速读取继续保留固定摘要卡，不为按需字段裁剪新增 Planner 协议和节点。

回归场景：

1. CRM 事实查询：不检索知识库、不运行领域 Agent。
2. 财务单项判断：语义选择知识库，召回财务规则，只运行财务 Agent。
3. 交付单项判断：召回上线周期和范围边界规则，只运行交付 Agent。
4. 合同单项判断：召回验收、退款、延期赔付规则，只运行合同 Agent。
5. 完整评审：
   - 四个领域 Agent 运行。
   - 不请求外部资料时不运行外部情报。
   - 不按四个完整章节重复。
   - 同一 CRM 事实最多完整解释一次。
   - 风险和优先级前后一致。
   - 规则依据可复核。
   - 包含一次责任边界。
6. 附件评审：附件事实与 CRM 事实并行进入检索和判断，冲突时明确要求人工确认。
7. 知识库空召回或失败：不伪造规则，仍能基于可用 CRM/附件证据给出有限判断。

继续记录：

- 所有超过 2 秒的节点。
- 各模型节点 Prompt、Completion 和 Total Tokens。
- 知识库检索耗时和返回条数。
- 检索决策 LLM 的输入、输出与耗时。

- 对 CRM 工作台公开的最终协议、`turnType`、`boundObject` 和 CRM Tool API 均不改变。
- 本计划不恢复真实 CRM 写回。

## 验收标准

- Dify 直接检索测试不再出现高度相关问题零召回。
- Planner 的知识库选择来自语义判断，代码中不存在关键词路由或硬评分。
- 基于 CRM 或附件事实生成的 `knowledge_query` 简洁、完整、无重复，不包含整包原始数据。
- 完整评审覆盖四领域价值，但不逐段复述四个 Agent。
- 输出没有明显事实重复、机械模板堆叠或越权审批表达。
- 风险和优先级由 LLM 综合判断，并在同一轮保持一致。
- 每个重要结论能追溯到 CRM、附件、外部来源或知识库规则；缺失证据时明确不确定性。
- 回答长度不设硬性字符限制，以信息价值、可读性和无赘述为准。
- 每个阶段完成后暂停，由用户导入、发布并验收后再进入下一阶段。
