# 🧾 输出解释层

本文件定义了如何将原始 JSON 响应转换为用户可读的信息。
所有 AI 生成的输出必须遵循本规范。

> 🔒 **安全提醒**：输出时禁止暴露 `CORDYS_ACCESS_KEY`、`CORDYS_SECRET_KEY` 或任何 API 凭证。如果 API 错误信息中包含密钥相关内容，必须先脱敏再展示。

---

## 1. 第一性原则

**输出不是汇报数据，是帮用户做判断。**

原始数据 → 你的分析 → 输出判断

每条输出应该遵循：
```
关键结论（如果有清晰发现）
└─ 核心数据（表格/列表，突出角色关注的字段）
   └─ 异常提醒（如果有）
      └─ 建议动作（如果有）
```

---

## 2. 格式化规则

### 数据展示

| 数据类型 | 格式 | 示例 |
|---------|------|------|
| 列表 | 表格，突出角色关注字段 | `│ 名称 │ 负责人 │ 金额 │ 状态 │` |
| 单条详情 | 关键字段摘要 + 其他字段折叠 | |
| 统计数据 | 总览 + 明细 | `本月合同 13 份，总金额 ¥285 万` |
| 错误/空 | 说明原因，不贴原始 JSON | `暂无数据（当前过滤条件下）` |

### 表格规则
- 使用标准 Markdown 表格（不要手工画 ASCII 框线）
- 列数 ≤ 5，多字段时分批展示
- 关键值（金额、日期、状态）放在前列

### Emoji 状态指示
| emoji | 含义 |
|-------|------|
| 🟢 | 正常、已完成、活跃 |
| ⚠️ | 需要注意、即将过期 |
| 🚨 | 紧急、逾期、异常 |
| ✅ | 已完成、已处理 |
| 📋 | 计划、提醒 |
| 📊 | 统计、概览 |

---

## 3. 角色感知的输出适配

根据角色调整展示优先级：

### 销售
- 优先展示：负责人、跟进时间、阶段、下一步动作
- 输出侧重：操作建议，具体到"联系谁 + 做什么 + 先后顺序"
- 摘要型列表优先，支持点击查看详情

### 经理
- 优先展示：团队统计、排名、覆盖率、目标进度
- 输出侧重：管理决策，结构化风险 + 建议
- 从宏观到微观：全貌 > 个人 > 具体记录

### 财务
- 优先展示：金额汇总、日期、发票状态、逾期天数
- 输出侧重：数值精确，金额必须带单位
- 异常项优先排列，正常项折叠

---

## 4. 大结果集处理策略

当一次查询返回较多记录时，按以下规则处理：

| 返回条数 | 展示方式 |
|---------|---------|
| 1-10 条 | 完整表格展示 |
| 11-30 条 | 展示前 10 条 + "还有 N 条，是否查看更多？" |
| 30 条以上 | 统计摘要 + 前 10 条 + "建议增加筛选条件" |

> **特别提醒**：30 条记录的 JSON 原始响应可能 2000+ tokens。**格式化输出成表格后，原始 JSON 在后续思考中已无保留价值**，不要反复引用它。

### 全局模糊搜索（6 模块并行）时的处理

这是最容易 token 暴涨的场景。正确的做法：

```
并行发起 6 个请求
  │
  每个请求返回后：
  ├─ 立即提取关键信息：命中总数、前 3-5 条记录的核心字段
  └─ 格式化表格存为摘要
      └─ 原始 JSON 已无用，不再引用
  │
  6 个模块全部完成后：
  └─ 合并各模块摘要，输出汇总报告
```

举例（不要这样）：
```
❌ 错误：保留 6 个原始 JSON 在思考中反复分析
    "线索模块返回了 JSON: {...大量字段...}，
     客户模块返回了 JSON: {...大量字段...}，
     然后从线索中找到...又从客户中找到..."
```

举例（应该这样）：
```
✅ 正确：提取摘要后直接输出
    线索: 8条
    客户: 2条 (华星科技)
    商机: 2条 (金额总计¥35万)
    → 输出跨模块表格 + "客户华星科技名下有关联线索和商机"
```

## 5. 输出反模式（禁止）

```
❌ 直接贴 JSON 响应
   {"code":100200,"data":{"list":[...],"total":13}}

❌ 纯搬运不做判断
   "您有 13 条线索，创建时间是···"

❌ 抛给用户选择但不给建议
   "有 3 条线索超期未跟进，您想先看哪个？"

❌ 表格列太多导致换行（超过 5 列分两批展示）
   "│ ID │ 名称 │ 负责人 │ 金额 │ 阶段 │ 创建时间 │ 跟进时间 │···"
```

## 7. 多模块搜索输出格式（全局模糊搜索）

当执行跨模块全局搜索（cli-spec.md §11）时，输出须遵循以下格式。

### 7.1 输出结构

```text
🔍 全局搜索："{搜索词}"
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📌 {模块中文名}（{命中条数} 条）
{数据表格或说明}

📌 {模块中文名}（{命中条数} 条）
{数据表格或说明}

...（所有有数据的模块按优先级排列；无数据的在最下方折叠）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 汇总：共找到 {总数} 条匹配记录
{根据角色给出的行动建议（如有）}
```

### 7.2 输出规则

| 规则 | 说明 |
|------|------|
| **有数据才展示** | 命中条数 = 0 的模块不展开表格，仅以单行显示无匹配 |
| **按优先级排序** | 线索、线索池、客户在前（高优），商机、公海居中，联系人在后 |
| **模块用 emoji 标记** | 📌 统一前缀，**不加模块名重复 emoji** |
| **表格 ≤5 列** | 按模块特性选择关键字段展示 |
| **汇总行** | 包含总命中数 |
| **行动建议** | 如果某个模块命中多且有明显趋势，给出操作建议 |
| **失败/超时** | 标注为 `⚠️ {模块} 查询失败` 或 `⚠️ {模块} 查询超时` |
| **关键词同名** | 如果多个模块命中同一条记录（如线索和客户都有"华星科技"），在建议中指出关联性 |

### 7.3 各模块表格列定义

| 模块 | 关键列（≤5 列） |
|------|----------------|
| 线索 | 名称 / 公司 / 电话 / 负责人 / 创建时间 |
| 线索池 | 名称 / 公司 / 电话 / 负责人 / 创建时间 |
| 客户 | 名称 / 行业 / 省份 / 负责人 / 创建时间 |
| 商机 | 名称 / 金额 / 阶段 / 负责人 / 创建时间 |
| 公海 | 名称 / 行业 / 省份 / 负责人 / 创建时间 |
| 联系人 | 姓名 / 公司 / 手机 / 邮箱 |

### 7.4 空结果处理

```text
📌 公海（0 条）
（无匹配结果）
```

### 7.5 失败/超时处理

```text
📌 线索池 ⚠️ 查询超时
📌 公海 ⚠️ 查询失败
```

### 7.7 行动建议规则

| 场景 | 建议内容 |
|------|---------|
| 线索 + 客户同时命中 | `💡 客户"{名称}"名下有关联线索和商机，建议进一步查看客户详情` |
| 线索池命中多 | `💡 线索池中有 {N} 条可领取线索，建议先看看` |
| 联系人命中多 | `💡 建议按公司/部门筛选联系人以缩小范围` |
| 仅 1 个模块有数据 | `💡 仅在 {模块} 中匹配到 {N} 条结果，是否需要扩大搜索范围？` |
| 无任何匹配 | `💡 未找到与"{搜索词}"相关的任何记录，请检查关键词是否正确` |
| 超过 30 条总结果 | `💡 搜索结果较多（{N} 条），建议增加筛选条件或指定模块` |
| 线索 + 客户 + 商机同时命中 | `💡 检测到完整链路：线索"{线索名}" → 客户"{客户名}" → 商机"{商机名}"（¥{金额}），建议查看客户360详情` |

              │
              ├── 🔹 商机 "{商机名}" ¥{金额} ({创建日期}, 阶段={阶段})

              │        ├── 📄 报价单 ({数量} 份)
              │        │
              │        └── 🔹 合同 {合同ID} ¥{金额} ({签约日期})
